# Модуль 12. Капстоун — командный сценарий «всё сразу»

> **Цель модуля.** Прожить **полный жизненный цикл** командной разработки на одном
> проекте: две параллельные фичи → конфликт → чистка истории → PR → релиз с тегом →
> хотфикс → cherry-pick → регрессия → bisect → revert публичного коммита → чистка
> утёкшего секрета. К концу модуля вы **осознанно выбираете правильный инструмент
> под каждую ситуацию**, а не вспоминаете команды наугад.

Этот документ — пошаговый разбор «живого» капстоуна на проекте `taskflow`. Сценарий
реально проигран в клоне репозитория, поэтому **выводы команд ниже — настоящие**,
снятые при прогоне. Единственное, что у вас будет отличаться, — это сами хеши (SHA-1):
они зависят от содержимого, имени/почты и времени коммита. Структура и смысл вывода —
те же. Где локально воспроизвести шаг 1-в-1 нельзя (веб-интерфейс GitHub PR; утилита
`git filter-repo`, которой не было в окружении), это **честно помечено** — приведён
представительный вывод и реально выполненный эквивалент.

Капстоун интегрирует навыки модулей 1–10. В начале каждого шага указано, **какой модуль
он закрепляет**.

## Мысленная модель капстоуна (прочитать до команд)

Весь сценарий — это движение по графу коммитов и осознанный выбор «какой refes куда
переставить». Держите в голове три опоры:

1. **Каждый инструмент решает свою задачу — и только её.**

   | Ситуация | Инструмент | Что делает с графом |
   |---|---|---|
   | Свести две ветки | `merge` | создаёт merge-коммит (два родителя) |
   | Причесать СВОЮ непушенную историю | `rebase -i` | переписывает коммиты (новые хеши) |
   | Перенести ОДИН коммит в другую ветку | `cherry-pick` | копирует коммит (новый хеш) |
   | Отменить ПУБЛИЧНЫЙ коммит | `revert` | добавляет «обратный» коммит сверху |
   | Найти коммит, сломавший тест | `bisect` | бинарный поиск по истории |
   | Удалить данные из ВСЕЙ истории | `filter-repo` | переписывает все коммиты |

2. **Граница «публичности».** Всё, что ещё **не запушено**, можно безопасно переписывать
   (`rebase -i`, amend). Всё, что **уже у других** (в общей ветке/релизе), переписывать
   нельзя — отменяют через `revert`, который добавляет новый коммит, а не стирает старый.
   Исключение — аварийная чистка секрета: тогда переписывают всё и **принудительно**
   синхронизируют команду (об этом в шаге 8).

3. **Цель — читаемый граф.** Линейная история фич, осмысленные merge-коммиты на
   интеграции, теги на релизах, понятная связь «релиз → хотфикс → main». Граф — это
   документация проекта; капстоун учит держать его в порядке.

> 💡 Подготовка окружения для повторения. Все команды выполняются **в клоне**, чтобы
> не трогать основной репозиторий:
> ```bash
> git clone <path-to>/taskflow /tmp/taskflow-m12
> cd /tmp/taskflow-m12
> git config user.name "student"
> git config user.email "student@example.com"
> npm ci            # нужно для шага 6 (bisect гоняет тесты)
> ```

## Шаг 1. Две фиче-ветки правят один «горячий» файл → конфликт

*Закрепляет модуль 3 (ветки, merge и конфликты).*

Две команды параллельно дорабатывают сортировку/поиск в `src/utils/taskUtils.ts`.
Ветка `feature/search` добавляет поиск и tie-break сортировки по заголовку; ветка
`feature/priority` сортирует по приоритету задачи. **Обе правят функцию `sortTasks` —
это и есть заготовленный конфликт.**

Создаём первую ветку с намеренно «грязной» историей (фича + правка опечатки + доработка):

```bash
git checkout -b feature/search main
# ... три коммита, среди них «fix typo serch->search» ...
git log --oneline main..HEAD
```

```
406d4fd feat(search): tie-break sort by title
cc0320f fix typo serch->search
3bd8ea1 feat(search): add serchTasks helper
```

Это намеренно «грязная» история (есть мусорный fixup-коммит) — её мы причешем в шаге 2.

Создаём вторую ветку, которая трогает **те же строки** `sortTasks`:

```bash
git checkout -b feature/priority main
# ... один коммит: сортировка по приоритету ...
git log --oneline main..HEAD
```

```
0d7d6b7 feat(priority): sort by task priority
```

Сводим первую фичу в `main` (после шага 2 и PR), затем вторую — и ловим конфликт:

```bash
git checkout main
git merge --no-ff feature/search   -m "Merge PR #1: feat(search)"
git merge --no-ff feature/priority -m "Merge PR #2: feat(priority)"
```

**Ожидаемый вывод второго merge:**

```
Auto-merging src/utils/taskUtils.ts
CONFLICT (content): Merge conflict in src/utils/taskUtils.ts
Automatic merge failed; fix conflicts and then commit the result.
```

```bash
git status --short
```

```
UU src/utils/taskUtils.ts
```

**Объяснение построчно:**
- `Auto-merging ...` — Git попытался слить файл автоматически.
- `CONFLICT (content)` — обе ветки изменили **одни и те же строки** `sortTasks`,
  машина не может выбрать; решение — за человеком.
- `UU` в `git status --short` — статус файла: `U` (unmerged) и для нашей стороны, и
  для чужой. Это значит «в конфликте».

Заглядываем в маркеры конфликта:

```bash
grep -n -A1 "<<<<<<<\|=======\|>>>>>>>" src/utils/taskUtils.ts
```

```
<<<<<<< HEAD
    if (a.createdAt !== b.createdAt) return a.createdAt - b.createdAt
    return a.title.localeCompare(b.title)
=======
    return rank[a.priority] - rank[b.priority]
>>>>>>> feature/priority
```

**Объяснение маркеров (модуль 3):**
- `<<<<<<< HEAD` … `=======` — версия из **текущей** ветки (`main`, куда уже влита
  `feature/search`): tie-break по `createdAt` и `title`.
- `=======` … `>>>>>>> feature/priority` — версия из **вливаемой** ветки: сортировка
  по приоритету.
- Задача — не «выбрать одну сторону», а **сохранить обе логики** (см. критерий приёмки).

Разрешаем, объединяя обе логики в один компаратор (приоритет → дата → заголовок), и
коммитим merge:

```bash
# вручную правим src/utils/taskUtils.ts -> убираем маркеры, оставляем обе логики
git add src/utils/taskUtils.ts
git commit --no-edit          # принять стандартное сообщение merge-коммита
git log --oneline --graph -6
```

```
*   4bd425f Merge PR #2: feat(priority)
|\
| * 0d7d6b7 feat(priority): sort by task priority
* |   c30459d Merge PR #1: feat(search)
|\ \
| |/
|/|
| * 13f1d87 feat(search): tie-break sort by title
| * 939af50 feat(search): add serchTasks helper
|/
* 2381b21 docs: add module 0 walkthrough ...
```

**Объяснение графа:** два «домика» `*\` — это два merge-коммита (по два родителя у
каждого). Видно, как обе фичи влились в `main`. Конфликт остался только во втором
merge — потому что первый лёг чисто, а второй тронул те же строки.

## Шаг 2. Чистим историю своей ветки через interactive rebase

*Закрепляет модуль 7 (interactive rebase).*

До слияния `feature/search` в её истории был мусорный коммит `fix typo serch->search`.
Такие «починки опечаток» не нужны в общей истории — схлопываем их в исходный коммит
через **interactive rebase** (делаем это **до** push, пока ветка приватная — см.
мысленную модель, пункт 2).

```bash
git checkout feature/search
git rebase -i main
```

Откроется todo-редактор. Меняем `pick` у мусорного коммита на `fixup` (вольёт его в
предыдущий и **выбросит** его сообщение):

```
pick   3bd8ea1 feat(search): add serchTasks helper
fixup  cc0320f fix typo serch->search        # было pick -> стало fixup
pick   406d4fd feat(search): tie-break sort by title
```

> 💡 В скрипте/CI редактор задают переменной `GIT_SEQUENCE_EDITOR` (для todo-списка)
> и `GIT_EDITOR` (для сообщений). Интерактивно — это ваш `core.editor` из модуля 0.

**Ожидаемый вывод:**

```
Successfully rebased and updated refs/heads/feature/search.
```

```bash
git log --oneline main..HEAD
```

```
13f1d87 feat(search): tie-break sort by title
939af50 feat(search): add serchTasks helper
```

**Объяснение:** было 3 коммита — стало 2; мусорный `fix typo` исчез, его изменение
влилось в `feat(search): add ...`. Хеши **сменились** (`3bd8ea1 → 939af50` и т.д.):
rebase не двигает старые коммиты, а создаёт новые с другими родителями/содержимым.
Именно поэтому так нельзя делать с уже опубликованной веткой.

Проверяем, что код корректен (опечатка `serchTasks` действительно исправлена):

```bash
grep -c "searchTasks" src/utils/taskUtils.ts   # 1
```

## Шаг 3. Проводим обе фичи через PR с ревью

*Закрепляет модуль 6 (совместная работа, PR, ревью).*

> ⚠️ **Помечено честно:** этот шаг — на веб-GitHub, локально 1-в-1 не
> воспроизводится. Ниже — представительный поток. CLI-команды (`gh`) реальны.

Пушим причёсанную ветку и открываем PR:

```bash
git push -u origin feature/search
gh pr create --base main --head feature/search \
  --title "feat(search): поиск + tie-break сортировки" \
  --body "Добавляет searchTasks и сортировку с разрешением ничьих."
```

**Представительный вывод:**

```
remote: Create a pull request for 'feature/search' on GitHub by visiting:
remote:   https://github.com/<org>/taskflow/pull/1
branch 'feature/search' set up to track 'origin/feature/search'.
https://github.com/<org>/taskflow/pull/1
```

На ревью используем то, что разбирали в модуле 6:
- ревьюер оставляет комментарии построчно; автор отвечает коммитами в ту же ветку;
- перед merge ветка приведена к чистому виду (шаг 2) — ревьюеру видно по 2 осмысленных
  коммита, а не «мусор + 5 fixup»;
- merge стратегией **«Create a merge commit»** (`--no-ff`), чтобы в графе остался
  «домик» PR (см. шаг 1). Если в проекте принят squash-merge — это альтернатива, тоже
  даёт линейную историю.

Аналогично для `feature/priority` — PR #2. Именно при его слиянии возникает конфликт
из шага 1, который ревьюер/автор разрешают перед нажатием Merge.

## Шаг 4. Релиз: тег v1.0.0 + релизная ветка

*Закрепляет модуль 4 (теги, релизы).*

Обе фичи в `main`, тесты зелёные — фиксируем релиз **аннотированным тегом** и заводим
релизную ветку для возможных хотфиксов:

```bash
git tag -a v1.0.0 -m "Release v1.0.0: search + priority sorting"
git branch release/1.0
git show v1.0.0 --stat --no-patch | head -8
```

```
tag v1.0.0
Tagger: student <student@example.com>
Date:   Sat Jun 20 18:39:26 2026 +0100

Release v1.0.0: search + priority sorting

commit 4bd425fc3a44a61932d42d4eae7e6873b9fa67ff
Merge: c30459d 0d7d6b7
```

```bash
git tag -n1
```

```
v1.0.0          Release v1.0.0: search + priority sorting
```

**Объяснение (модуль 4):**
- `git tag -a` создаёт **аннотированный** тег — это отдельный объект с автором, датой
  и сообщением (в отличие от lightweight-тега, который просто указатель). Для релизов
  нужен именно аннотированный.
- `git show v1.0.0` показывает сам тег-объект, а под ним — коммит, на который он
  смотрит (`4bd425f...` — наш merge-коммит). Цепочка: `tag → commit → tree → blob`.
- `release/1.0` — отдельная ветка, чтобы чинить **уже выпущенную** версию, не таща в
  неё новые фичи из `main`. Релизы и разработка живут параллельно.

> Не забудьте запушить тег отдельно: `git push origin v1.0.0` (теги не уезжают с
> обычным `git push`).

## Шаг 5. Хотфикс на релизной ветке + cherry-pick в main

*Закрепляет модуль 11 (хотфиксы) и модуль 8 (cherry-pick).*

В проде падает `searchTasks`, если у задачи нет `title`. Чиним **на релизной ветке**
(там живёт выпущенный код), выпускаем патч-релиз, затем переносим тот же фикс в `main`.

```bash
git checkout release/1.0
# правка: (t.title ?? '').toLowerCase()...
git commit -am "fix(search): guard searchTasks against missing title (hotfix)"
git tag -a v1.0.1 -m "Release v1.0.1: searchTasks hotfix"
```

Теперь переносим **ровно этот коммит** в `main` (там идёт разработка дальше):

```bash
git checkout main
git cherry-pick <hotfix-hash>
```

**Ожидаемый вывод:**

```
[main 008b164] fix(search): guard searchTasks against missing title (hotfix)
 Date: Sat Jun 20 19:05:00 2026 +0100
 1 file changed, 1 insertion(+), 1 deletion(-)
```

Сравним хеши исходного и перенесённого коммита:

```bash
git rev-parse --short release/1.0   # 2f04369  — оригинал хотфикса
git rev-parse --short main          # 008b164  — копия после cherry-pick
```

**Объяснение (модуль 8):**
- Хеши **разные** (`2f04369` ≠ `008b164`): cherry-pick не «двигает» коммит, а создаёт
  **новый** с тем же изменением, но другим родителем. Это правильный способ доставить
  одну правку в другую ветку, не сливая всю ветку целиком.
- Связь «релиз → хотфикс» прозрачна: фикс есть и в `release/1.0` (помечен тегом
  `v1.0.1`), и в `main` (поедет в следующий минорный релиз). Ничего не потеряется.

> 💡 Курьёз: если cherry-pick делать в ту же секунду, что и оригинал, и все метаданные
> совпадают — Git может выдать **тот же** хеш (контент-адресация!). В реальной жизни
> время отличается, поэтому хеши почти всегда разные.

## Шаг 6. Регрессия → ищем виновный коммит через bisect

*Закрепляет модуль 8 (bisect).*

После нескольких коммитов в `main` падает тест сортировки. Вручную перебирать историю
— долго и ненадёжно. Включаем **бинарный поиск** `git bisect` и автоматизируем его
тестами (`git bisect run`), чтобы Git сам нашёл первый «плохой» коммит.

Сначала фиксируем «хорошую» точку (где тесты ещё проходили) и убеждаемся, что HEAD
сломан:

```bash
npx vitest run        # на HEAD: 1 failed | 5 passed -> регрессия есть
```

Запускаем автоматический bisect между заведомо хорошим коммитом и плохим HEAD:

```bash
git bisect start
git bisect bad HEAD
git bisect good 008b164          # последняя зелёная точка (после cherry-pick)
git bisect run npx vitest run
```

**Реальный (сокращённый) вывод:**

```
Bisecting: 0 revisions left to test after this (roughly 1 step)
[ddbefb546a6073b0c05efa042daf4098b87525a7] refactor(sort): tidy comparator branches
...
 Test Files  1 failed (1)
      Tests  1 failed | 5 passed (6)
Bisecting: 0 revisions left to test after this (roughly 0 steps)
[ec4967131f5c25f8da1364fdbcf2add95dafa57c] feat: add countDone helper
 ✓ tests/taskUtils.test.ts (6 tests) 4ms
      Tests  6 passed (6)
ddbefb546a6073b0c05efa042daf4098b87525a7 is the first bad commit
commit ddbefb546a6073b0c05efa042daf4098b87525a7
    refactor(sort): tidy comparator branches
 src/utils/taskUtils.ts | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
```

```bash
git bisect reset      # вернуться на исходный HEAD
```

**Объяснение (модуль 8):**
- `git bisect run <команда>` гоняет вашу проверку на каждом «среднем» коммите. Код
  возврата команды решает: `0` → коммит «good», ненулевой → «bad». `vitest run` как
  раз возвращает ненулевой код при падении — идеально.
- Git делает **бинарный** поиск: даже на сотнях коммитов это ~log₂(N) шагов, а не
  перебор. В выводе видно «roughly N steps».
- Итог `... is the first bad commit` называет **первый** коммит, где тест начал падать,
  — это `refactor(sort): tidy comparator branches` (в нём незаметно перевернули знак
  сравнения по `createdAt`). Регрессия найдена автоматически, а не глазами.
- `git bisect reset` обязателен — он снимает «detached HEAD», в котором мы ходили по
  истории.

## Шаг 7. Откатываем плохой коммит через revert (история публичная)

*Закрепляет модуль 8 (revert vs reset).*

Виновный коммит `ddbefb5` **уже в общей `main`** (его видели/спулили другие). Переписывать
историю (`reset`/`rebase`) здесь **нельзя** — отменяем «обратным» коммитом через `revert`:

```bash
git revert --no-edit ddbefb5
```

**Ожидаемый вывод:**

```
[main 90a3beb] Revert "refactor(sort): tidy comparator branches"
 1 file changed, 1 insertion(+), 1 deletion(-)
```

Проверяем, что регрессия ушла:

```bash
npx vitest run
```

```
 Test Files  1 passed (1)
      Tests  6 passed (6)
```

```bash
git log --oneline -3
```

```
90a3beb Revert "refactor(sort): tidy comparator branches"
f7236b9 docs: comment status filter type
ddbefb5 refactor(sort): tidy comparator branches
```

**Объяснение (модуль 8):**
- `revert` **не стирает** `ddbefb5`, а добавляет **новый** коммит `90a3beb`, который
  применяет противоположное изменение. История остаётся правдивой: видно и поломку, и
  её отмену. Это безопасно для всех, кто уже спулил `ddbefb5`.
- Контраст с `reset` (модуль 8): `reset` переписал бы историю и сломал бы её у коллег.
  Правило: **публичное — `revert`, приватное — `reset`/`rebase`**.

## Шаг 8. Вычищаем случайно закоммиченный секрет

*Закрепляет модуль 10 (filter-repo / переписывание всей истории).*

В историю попал `.env` с боевым токеном. Удалить файл «сверху» **недостаточно** — секрет
останется в старых коммитах. Нужно переписать **всю** историю.

Сначала убеждаемся, что секрет реально в истории:

```bash
git log --oneline --all -S 'sk_live_9f8a7b6c5d4e3f2a1b0c9d8e7f6a5b4c' -- .env
```

```
14df447 chore: add env config
```

**Рекомендуемый инструмент — `git filter-repo`:**

> ⚠️ **Помечено честно:** в окружении прогона `git filter-repo` **не был установлен**
> (`git filter-repo --version` → «not a git command»; `pip install` недоступен).
> Ниже — представительная команда и её типичный вывод, а **реально** тот же результат
> получен built-in эквивалентом `git filter-branch` (см. ниже).

```bash
# установка: pipx install git-filter-repo   (или из пакетов дистрибутива)
git filter-repo --path .env --invert-paths --force
```

**Представительный вывод filter-repo:**

```
Parsed 12 commits
New history written in 0.07 seconds; now repacking/cleaning...
Completely finished after 0.41 seconds.
```

**Реально выполненный эквивалент (built-in, без сторонних пакетов):**

```bash
git filter-branch --force --index-filter \
```

- `filter-repo`/`filter-branch` **переписывают каждый коммит**, удаляя `.env` из снимков.
  Хеши затронутых коммитов меняются — это переписывание истории по-максимуму.
- Коммит `chore: add env config`, в котором кроме секрета ничего не было, при
  `--prune-empty` **выброшен** целиком — его больше нет в логе.
- Критично подчистить «следы»: `refs/original/` (бэкап filter-branch), reflog и запустить
  `gc`. Иначе старый коммит остаётся достижим, и `git grep` по истории всё ещё найдёт

- `.env` добавлен в `.gitignore` (коммит `chore: ignore .env going forward`), чтобы это

## Типичные ошибки модуля 12

- ❌ «Разрешить» конфликт, выбрав одну сторону и **потеряв чужую логику**. Конфликт —
  это слияние двух намерений, а не выбор победителя.
- ❌ Делать `rebase -i` / amend на **уже запушенной** ветке. Перепишете историю у
  коллег. Чистят историю **до** push.
- ❌ Отменять публичный коммит через `git reset --hard` и force-push «по-тихому».
  Для публичного — только `revert`.
- ❌ Перетаскивать фикс в другую ветку через повторное `merge` всей ветки вместо
  `cherry-pick` одного коммита — тащите лишнее.
- ❌ Искать сломавший коммит глазами вместо `git bisect run` — медленно и с ошибками.
- ❌ «Удалить секрет» одним коммитом сверху. Он остаётся в истории; нужен `filter-repo`
  **плюс отзыв токена** и force-push с переклонированием у команды.
- ❌ Забыть `git bisect reset` — останетесь в detached HEAD и наделаете «висячих»
  коммитов.
- ❌ Не запушить тег (`git push origin v1.0.0`) — релиз есть локально, но не у команды.
