# Модуль 2. Гигиена репозитория: что не коммитят

> **Цель модуля.** Понять, какие файлы **не должны** попадать в историю (зависимости,
> артефакты сборки, секреты, мусор), уверенно настроить **`.gitignore`** и понимать его
> синтаксис, знать про главный подвох «`.gitignore` не действует на уже отслеживаемое»
> и уметь вычистить такие файлы через **`git rm --cached`**, а также настроить
> **`.gitattributes`** для нормализации окончаний строк.

Этот документ — продолжение живого примера на проекте `taskflow`. В модуле 0 мы
специально сделали первый коммит **до** `npm install`, а в модуле 1 освоили базовый
цикл на чистых исходниках. Теперь доведём проект до рабочего состояния (поставим
зависимости, соберём билд) — и заодно разберём самую частую ошибку новичка: «закоммитить
всё подряд».

---

## Мысленная модель (прочитать до команд)

Как только вы запускаете проект по-настоящему, рядом с исходниками появляются файлы,
которым **не место в истории**. Их объединяет одно свойство:

> **Правило: в Git идёт только то, что нельзя восстановить из другого закоммиченного.**
> Всё, что генерируется командой (`npm install`, `npm run build`), — за бортом.

И два инструмента гигиены делят зоны ответственности:

- **`.gitignore`** решает, **что** вообще попадает в репозиторий (какие файлы Git «не
  видит»).
- **`.gitattributes`** решает, **как** Git **обрабатывает** содержимое уже
  отслеживаемых файлов (в первую очередь — окончания строк).

---

## Что вообще не коммитят и почему

- **`node_modules/`** — зависимости. Их восстанавливает `npm install` из
  `package.json` + `package-lock.json`. Это десятки тысяч чужих файлов: они раздувают
  репозиторий, ломают `diff` и привязаны к ОС.
- **`dist/`, `build/`, `*.tsbuildinfo`** — артефакты сборки. Воспроизводятся командой
  `npm run build`. Хранить их в Git — то же, что коммитить скомпилированный `.exe`.
- **`.env`, `*.local`** — локальные настройки и **секреты**. В историю не должны
  попадать никогда (как вычищать уже утёкший секрет — отдельная большая тема модуля 12).
- **`.DS_Store`, `Thumbs.db`, `*.log`** — системный и runtime-мусор.

---

## Шаг 1. Воспроизводим ошибку (чтобы знать, как чинить)

Устанавливаем зависимости и собираем проект:

```bash
npm install
npm run build
```

Теперь в рабочей директории появились `node_modules/` и `dist/`. Смотрим статус:

```bash
git status --short
```

**Вывод:**

```
?? dist/
?? node_modules/
?? tsconfig.tsbuildinfo
```

Git **сворачивает** целые неотслеживаемые папки в одну строку (`node_modules/`), хотя
внутри тысячи файлов. И вот здесь новичок по привычке делает «закоммитить всё»:

```bash
git add -A
git commit -m "chore: add dependencies and build"
```

**Вывод:**

```
[main 9f3c1a2] chore: add dependencies and build
 12473 files changed, 1894521 insertions(+)
 create mode 100644 dist/assets/index-4af1b2.js
 create mode 100644 node_modules/.bin/acorn
 ...
```

**`12473 files changed`** — вот она, катастрофа. В историю навсегда вшились чужие
зависимости. Дальше каждый `git clone` тащит этот балласт, а `git log` по файлам
тонет в шуме.

## Шаг 2. `.gitignore` — правила, по которым Git «не видит» файлы

Создаём в корне репозитория файл `.gitignore`:

```gitignore
# Зависимости
node_modules/

# Результат сборки
dist/
*.tsbuildinfo

# Локальное окружение и секреты
.env
.env.local
*.local

# Логи
*.log
npm-debug.log*

# Системный мусор
.DS_Store
Thumbs.db
```

**Как читаются правила (синтаксис):**

| Шаблон          | Что значит                                                                           |
| --------------- | ------------------------------------------------------------------------------------ |
| `node_modules/` | Папка целиком (слеш в конце = только директория, не файл с таким именем).            |
| `*.log`         | Любой файл, оканчивающийся на `.log`, в любой папке. `*` — любые символы, кроме `/`. |
| `/dist`         | Ведущий `/` **привязывает к корню** репозитория: только `./dist`, не `src/dist`.     |
| `**/tmp`        | `**` — любое число вложенных папок: `tmp` на любой глубине.                          |
| `!keep.log`     | `!` — **исключение из правила**: этот файл не игнорировать, даже если выше `*.log`.  |
| `# текст`       | Комментарий. Пустые строки игнорируются.                                             |

Порядок важен: **выигрывает последнее совпавшее правило** — поэтому `!`-исключения
ставят после общего шаблона.

> ⚠️ **Главный подвох.** `.gitignore` действует только на **неотслеживаемые** файлы.
> Если файл **уже закоммичен** (а мы только что закоммитили `node_modules/`), новое
> правило его **не выкинет** — Git продолжит его отслеживать. Сначала надо перестать
> его трекать.

## Шаг 3. `git rm --cached` — перестаём отслеживать уже закоммиченное

Убираем артефакты из индекса, **оставляя их на диске**:

```bash
git rm -r --cached node_modules dist tsconfig.tsbuildinfo
```

**Вывод:**

```
rm 'node_modules/.bin/acorn'
rm 'dist/assets/index-4af1b2.js'
...
```

- `--cached` — ключевой флаг: удалить **только из индекса (и из будущих коммитов)**,
  файлы в working tree не трогать. Без `--cached` команда стёрла бы их и с диска.
- `-r` — рекурсивно, для папок.

Проверяем статус:

```bash
git status --short
```

**Вывод:**

```
A  .gitignore
D  dist/assets/index-4af1b2.js
D  node_modules/.bin/acorn
...
```

Файлы помечены на удаление из репозитория (`D`), а `node_modules/`/`dist/` больше **не**
показываются как untracked — теперь их прячет `.gitignore`. Коммитим чистку:

```bash
git add .gitignore
git commit -m "chore: stop tracking deps and build artifacts, add .gitignore"
```

Проверяем, что в истории артефактов больше нет:

```bash
git ls-files | grep -E "node_modules|dist/" | head
```

**Вывод:** пусто. `git ls-files` показывает только **отслеживаемые** файлы — раз grep
ничего не нашёл, балласт вычищен из индекса. (Из _истории_ старый коммит он не убирает —
коммит `9f3c1a2` с 12473 файлами всё ещё в прошлом; как переписывать историю,
будет в модулях 7 и 9. Но дальше расти репозиторий не будет.)

## Шаг 4. `.gitattributes` — нормализация контента

`.gitignore` решает, _что_ попадает в репозиторий. `.gitattributes` решает, _как_ Git
**обрабатывает** содержимое отслеживаемых файлов — в первую очередь окончания строк.
Помните `core.autocrlf` из модуля 0? Это **глобальная** настройка машины, и у каждого
участника она своя. `.gitattributes` **лежит в репозитории** и действует на всех
одинаково — поэтому он надёжнее.

Создаём `.gitattributes` в корне:

```gitattributes
# По умолчанию: Git сам определяет текстовые файлы и хранит их с LF
* text=auto eol=lf

# Скрипты — строго LF, иначе сломаются на Unix даже при правке с Windows
*.sh text eol=lf

# Бинарные файлы: не конвертировать и не пытаться делать текстовый diff
*.png binary
*.jpg binary
*.ico binary

# Lock-файл: помечаем сгенерированным — не «шумит» в diff и сворачивается в review на GitHub
package-lock.json linguist-generated=true -diff
```

**Что это даёт:**

- `* text=auto eol=lf` — в репозитории все текстовые файлы хранятся с **LF**, независимо
  от ОС автора. Это лечит «фантомные» диффы, где весь файл «изменён» только из-за
  переключения CRLF↔LF между Windows и Unix.
- `binary` — короткая запись для `-text -diff`: Git не трогает байты и не печатает
  бессмысленный diff по картинке.
- `linguist-generated` / `-diff` — на GitHub такие файлы сворачиваются в diff и не
  учитываются в языковой статистике репозитория.

Коммитим:

```bash
git add .gitattributes
git commit -m "chore: add .gitattributes for line-ending normalization"
```

> 💡 Если в репозитории уже были файлы с «неправильными» окончаниями, после добавления
> `.gitattributes` их нормализуют разово: `git add --renormalize .` и коммит.

Итог: рабочая директория снова чистая, `node_modules/`/`dist/` под игнором, окончания
строк зафиксированы для всей команды. **Это базовая гигиена, которую настраивают в
самом начале любого проекта** — мы сделали её здесь намеренно «задним числом», чтобы
показать и ошибку, и лечение.

## Типичные ошибки модуля 2

- ❌ **Коммит артефактов и зависимостей** (`node_modules/`, `dist/`, `*.tsbuildinfo`) —
  раздувает репозиторий и шумит в истории. Их место в `.gitignore`.
- ❌ **Думать, что `.gitignore` уберёт уже закоммиченный файл.** Не уберёт — для этого
  нужен `git rm --cached`. Правило срабатывает только на ещё неотслеживаемые файлы.
- ❌ **`git rm` без `--cached`** — сотрёт файл и с диска, а не только из индекса.
- ❌ **Закоммитить `.env` с секретами.** `.gitignore` спасает только до первого коммита;
  уже утёкший секрет придётся вычищать из всей истории (модуль 12) и менять.
- ❌ **Игнорировать окончания строк в команде** → «весь файл изменился» в diff на ровном
  месте. Лечится `.gitattributes` (`* text=auto eol=lf`) на уровне репозитория.

## Шпаргалка команд модуля 2

```bash
# что реально отслеживается / почему файл игнорится
git ls-files                  # список отслеживаемых файлов

git check-ignore -v <path>    # какое правило .gitignore прячет файл

# чистка уже закоммиченного
git rm -r --cached <path>     # перестать отслеживать (файлы остаются на диске)
git rm --cached <file>        # то же для одного файла

# нормализация окончаний строк
git add --renormalize .       # пересохранить файлы по правилам .gitattributes
```

## Домашнее задание

На своём клоне `taskflow` (или на любом своём проекте) наведите гигиену с нуля.

### Что нужно сделать

1. Убедитесь, что `node_modules/` и `dist/` существуют (`npm install && npm run build`).
2. **Сознательно** закоммитьте артефакт — например, `git add dist && git commit -m "oops"` —
   и посмотрите в `git status`/`git ls-files`, что он попал в индекс.
3. Напишите `.gitignore`, закрывающий зависимости, билд, логи, секреты и системный мусор.
4. Вычистите уже закоммиченный артефакт через `git rm -r --cached` и закоммитьте чистку.
5. Добавьте `.gitattributes` с `* text=auto eol=lf` и хотя бы одним правилом `binary`.

### Критерии «сделано»

- [ ] `git ls-files | grep -E "node_modules|dist"` — **пусто**.
- [ ] `git check-ignore -v dist` показывает сработавшее правило из `.gitignore`.
- [ ] В репозитории есть закоммиченные `.gitignore` и `.gitattributes`.
- [ ] `git status` — `working tree clean`, при этом `node_modules/`/`dist/` физически на диске.

### Подсказки

- Если случайно сделали `git rm` без `--cached` и файлы пропали с диска — не паникуйте:
  `git restore <path>` вернёт их из индекса/HEAD (а зависимости проще переустановить).
- `git check-ignore -v <path>` — лучший способ понять, **какое именно** правило (и из
  какой строки `.gitignore`) прячет файл, когда их несколько.
