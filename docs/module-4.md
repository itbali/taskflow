# Модуль 4. Теги — релизные метки

> **Цель модуля.** Понять, что тег — это **неподвижный** указатель на коммит (в отличие
> от ветки), различать **лёгкий** и **аннотированный** теги и осознанно выбирать второй
> для релизов, уметь ставить/смотреть/удалять/переносить теги и помечать тегом старый
> коммит. Попутно — базовая идея **semantic versioning**. (Публикация тегов на удалёнку —
> в модуле 5, вместе с `push`; здесь всё **локально**.)

Этот документ — живой пример на проекте `taskflow`. К этому моменту в `main` уже есть
готовая фича из модуля 1 — «отметка выполненных задач». Самое время зафиксировать её
как первый релиз и разобраться с тегами целиком.

---

## Мысленная модель (прочитать до команд)

В модуле 3 мы выяснили, что **ветка — это подвижный указатель на коммит**: делаешь
коммит — ветка едет вперёд. **Тег — это указатель, который НЕ двигается.** Поставили
`v1.0.0` на коммит — он навсегда показывает именно на этот коммит, что бы ни происходило
с ветками дальше.

```
ветка main:   ●───●───●───●   ← main едет вперёд с каждым коммитом
                  ↑
                v1.0.0          ← тег «прибит» к коммиту и стоит на месте
```

Зачем это нужно: **релиз — это «коммит, на который указывает тег `vX.Y.Z`».** Тег даёт
стабильное имя версии, по которому всегда можно вернуться к точному состоянию кода
(«что именно мы выкатили в 1.0.0?»).

Два вида тегов:

- **Лёгкий (lightweight)** — просто имя-указатель на коммит, без метаданных. Как ветка,
  только неподвижная.
- **Аннотированный (annotated)** — полноценный **объект Git** (помните четвёртый тип из
  модуля 0: `blob/tree/commit/tag`?) со своим автором, датой и сообщением. Для релизов
  используют **только** аннотированные.

## Шаг 1. Ставим первый релизный тег — `git tag -a`

Фича готова — помечаем версию аннотированным тегом:

```bash
git tag -a v0.1.0 -m "Релиз: отметка выполненных задач"
git tag
git log --oneline --graph --decorate --all
```

**Вывод:**

```
v0.1.0
```

```
* 166bed4 (HEAD -> main, tag: v0.1.0) feat: show "выполнено" badge for done tasks
* baceabb feat: highlight done task with aria-checked
* 6061d7c chore: initial taskflow scaffold (Vite + React + TS)
```

**Объяснение:**

- `git tag -a v0.1.0 -m "..."` создаёт **аннотированный** тег — отдельный объект со своим
  автором, датой и сообщением, указывающий на коммит.
- В `git log --decorate` появился `tag: v0.1.0` рядом с `HEAD -> main` — тег «прибит»
  к коммиту `166bed4` и больше не двигается (в отличие от ветки).
- `git tag` без аргументов — список всех тегов.

## Шаг 2. Что внутри тега — `git show` и `cat-file`

```bash
git show v0.1.0
```

```
tag v0.1.0
Tagger: itbali <xopycaku@gmail.com>
Date:   Sat Jun 20 18:35:54 2026 +0100

Релиз: отметка выполненных задач

commit 166bed41217d5658e419a68231d824b373e3ab4f
...
```

Сверху — данные **самого тег-объекта** (`Tagger`, дата, сообщение), ниже — коммит, на
который он указывает, и его diff. Убедимся, что это действительно отдельный объект:

```bash
git cat-file -t v0.1.0      # tag
git cat-file -t v0.1.0^{}   # commit  (^{} = «разыменуй тег до коммита»)
```

Аннотированный тег — это объект типа `tag`, который ссылается на `commit`. Лёгкий тег
таким объектом **не** является — он сразу указывает на коммит.

---

## Шаг 3. Лёгкий vs аннотированный — в чём разница на практике

```bash
git tag tmp-light            # лёгкий тег (без -a/-m)
git cat-file -t tmp-light    # commit  ← это сразу коммит, отдельного объекта нет
git tag -d tmp-light         # удаляем, он нам не нужен
```

| | Лёгкий | Аннотированный (`-a`) |
| --- | --- | --- |
| Что это | имя → коммит | отдельный объект `tag` → коммит |
| Автор/дата/сообщение | нет | есть |
| `git show` | показывает только коммит | показывает метаданные тега + коммит |
| Для чего | временная локальная закладка | **релизы** |

**Правило:** для всего, что увидят другие (релизы, версии), — `git tag -a`. Лёгкий тег —
только «для себя на пять минут».

## Шаг 4. Тег на старый коммит и список с фильтром

Тег можно поставить не только на `HEAD`, но и на **любой** коммит по хешу:

```bash
git tag -a v0.0.9 baceabb -m "Пред-релиз: подсветка выполненных"
```

Смотрим теги с сообщениями и по маске:

```bash
git tag -n                   # список тегов + первая строка сообщения
git tag -l "v0.1.*"          # только теги, подходящие под шаблон
```

**Вывод `git tag -n`:**

```
v0.0.9          Пред-релиз: подсветка выполненных
v0.1.0          Релиз: отметка выполненных задач
```

---

## Шаг 5. Удаление и перемещение тега

```bash
git tag -d v0.0.9                          # удалить локальный тег
git tag -fa v0.1.0 <new-hash> -m "..."     # -f = передвинуть тег на другой коммит
```

> ⚠️ Перемещать уже **опубликованный** тег — плохая практика: у тех, кто его скачал,
> останется старая версия. Релизные теги должны быть **иммутабельными**: ошиблись —
> выпускайте новый (`v0.1.1`), а не переставляйте старый.

---

> 📌 **Публикация.** Всё в этом модуле — локально: теги пока живут только в вашем
> репозитории. На удалёнку (`git push origin --follow-tags`) они уезжают **отдельно** от
> коммитов — это разберём в модуле 5, где появляется `push`.

---

## Про версии: semantic versioning (коротко)

Релизные теги принято именовать по схеме **`vMAJOR.MINOR.PATCH`** (semver):

- **MAJOR** (`v2.0.0`) — несовместимые изменения (ломают тех, кто зависит от вас);
- **MINOR** (`v1.1.0`) — новая функциональность, обратно совместимая;
- **PATCH** (`v1.0.1`) — багфиксы без новой функциональности.

Дальше в курсе (модуль 11) теги `vX.Y.Z` лягут в основу релизных стратегий — Git Flow,
release-ветки, хотфиксы.

## Типичные ошибки модуля 4

- ❌ **Лёгкий тег для релиза.** Без автора/даты/сообщения непонятно, кто и когда выпустил
  версию. Для релизов — всегда `git tag -a`.
- ❌ **Переставлять уже опубликованный тег** (`-f`) — у скачавших останется старая версия.
  Иммутабельность релиза важнее.
- ❌ **Путать тег и ветку.** Тег не двигается; «коммитить в тег» нельзя. Нужна подвижная
  метка — это ветка.

## Шпаргалка команд модуля 4

```bash
# создать
git tag -a vX.Y.Z -m "..."        # аннотированный (для релизов)
git tag vX.Y.Z                     # лёгкий (временная закладка)
git tag -a vX.Y.Z <hash> -m "..."  # тег на конкретный коммит

# смотреть
git tag                            # список тегов
git tag -n                         # список + сообщения
git tag -l "v1.*"                  # фильтр по маске
git show vX.Y.Z                    # метаданные тега + коммит

# удалять / двигать
git tag -d vX.Y.Z                  # удалить локальный тег
git tag -fa vX.Y.Z <hash> -m "..." # передвинуть (осторожно!)

# публикация тегов на удалёнку — в модуле 5 (git push --follow-tags)
```
