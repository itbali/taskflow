# Модуль 7. Переписывание истории: rebase

> **Цель модуля.** Научиться держать историю чистой и линейной **осознанно, а не из
> страха**. К концу модуля вы своими словами объясняете разницу **merge vs rebase**,
> знаете **золотое правило** (не переписывать опубликованную/общую историю) и умеете
> безопасно форс-пушить через `--force-with-lease`.

Этот документ — пошаговый разбор «живого примера» модуля 7 на проекте `taskflow`.
Все выводы команд ниже — **настоящие**, снятые на реальном репозитории. При
переписывании истории Git создаёт **новые** коммиты (с новыми хешами), поэтому у вас
SHA-1 будут другими — это абсолютно нормально и не означает ошибку. Структура графа,
тексты сообщений Git и смысл вывода — те же.

## Мысленная модель (прочитать до команд)

### merge vs rebase: что происходит с графом

Обе команды решают одну задачу — «соединить работу двух веток», — но делают это
по-разному.

**`merge`** берёт две ветки и создаёт **новый коммит со сливаемыми (двумя) родителями**.
История сохраняется как есть, со всеми ветвлениями. Граф нелинейный, зато правдивый:
видно, _когда реально_ ответвлялись и сливались.

**`rebase`** берёт ваши коммиты, **«отрывает» их от старой базы и переставляет поверх
другого коммита**, переписывая каждый заново. Старые коммиты не меняются — создаются их
**копии с новыми хешами**, а ветка-указатель переезжает на верхушку новой цепочки.
Граф получается линейным, как будто вы начинали работать уже от свежего `main`.

Наглядно. Было (ветка `feature` ответвилась, а `main` ушёл вперёд):

```
      A───B   feature
     /
*───*───C     main
```

После `git merge main` в feature:

```
      A───B───M   feature   (M — merge-commit, 2 родителя: B и C)
     /       /
*───*───C───┘     main
```

После `git rebase main` в feature:

```
          A'──B'   feature   (A',B' — НОВЫЕ копии A,B)
         /
*───*───C        main
```

Ключевое: при `rebase` коммиты `A` и `B` исчезают из ветки и заменяются на `A'`/`B'`
с другим содержимым метаданных (другой parent → другой хеш). Это и есть «переписывание
истории».

### Interactive rebase — это «редактор пачки коммитов»

`git rebase -i <база>` открывает **todo-список** ваших коммитов (от старого к новому,
сверху вниз) и позволяет к каждому применить операцию:

- `pick` — взять как есть;
- `reword` — взять, но **переписать сообщение**;
- `edit` — взять, но **остановиться** на нём (поправить файлы / разбить);
- `squash` — **слить** с предыдущим, **объединив сообщения**;
- `fixup` — слить с предыдущим, **выбросив** сообщение этого коммита;
- `drop` — **выкинуть** коммит целиком;
- **переставить строки** = **переупорядочить** коммиты.

Git перепроигрывает коммиты по этому списку и собирает новую цепочку.

> **`squash` vs `fixup` — в чём разница.** Обе операции **склеивают** коммит с тем, что
> стоит **выше** в todo (с предыдущим). Отличие только в **судьбе сообщения**:
> - `squash` — Git откроет редактор, где **оба сообщения** лежат вместе, и вы соберёте из
>   них одно (объединить, переписать — как угодно). Берут, когда сообщение склеиваемого
>   коммита **несёт смысл**.
> - `fixup` — сообщение склеиваемого коммита **молча выбрасывается**, остаётся только
>   сообщение «коммита-цели» сверху. Берут для мелких правок-«дописок» (`oops`,
>   `fix typo`, `again`), у которых отдельное сообщение в истории не нужно.
>
> Проще говоря: **`fixup` = `squash`, но без диалога про сообщение** — «подшей эту правку
> к тому коммиту и забудь её текст». Содержимое (diff) в обоих случаях сохраняется
> полностью — выбрасывается только текст сообщения.

### Золотое правило rebase

> **Не переписывайте историю, которую уже видят другие.**

Переписывать можно свободно **свои локальные** и **свою личную PR-ветку**, которую,
кроме вас, никто не разрабатывает. Нельзя `rebase` общих веток (`main`, `develop`,
релизных): вы создадите _параллельную_ историю, и у коллег при следующем `pull`
начнётся хаос со слияниями и дублями коммитов.

### Безопасный force-push

После rebase локальная ветка **расходится** с её копией на сервере (хеши-то новые),
и обычный `git push` будет отклонён как non-fast-forward. Нужен форс. Но:

- `git push --force` (короткая форма — **`git push -f`**) — «затолкать поверх, что бы
  там ни было». Именно `-f` по привычке набирают чаще всего — и именно она опасна: если
  коллега тем временем добавил в эту ветку коммит, вы его **молча затрёте**.
- `git push --force-with-lease` — «затолкать, **только если** серверная ветка всё ещё
  там, где я думаю». Если кто-то успел запушить — Git **откажет**, и вы ничего не
  потеряете. Короткой буквенной формы у неё нет (и это к лучшему — её набирают
  осознанно). Всегда используйте именно `--force-with-lease`, а не `-f`/`--force`.

> 💡 Страх «потерять коммиты» при rebase почти всегда напрасен: старые коммиты ещё
> какое-то время живут в **reflog** и достижимы по хешу. Это тема модуля 8 — там
> научимся «доставать» их обратно.

## Шаг 0. Готовим ветку

Работаем прямо в своём `taskflow` — удалёнку `origin` подключили ещё в модуле 5.
Всё переписывание делаем в **отдельной фиче-ветке**, а не в `main`: эта ветка личная,
её, кроме вас, никто не разрабатывает — поэтому переписывать её историю безопасно (это и
есть условие из золотого правила). `main` мы не трогаем вовсе.

Создадим ветку `feature/task-search` и наделаем в ней **6 «грязных» коммитов** — ровно
так, как это выглядит в реальной работе «по ходу мысли»:

```bash
git switch -c feature/task-search
# ... правим src/utils/taskUtils.ts ...
git commit -m "wip"
git commit -m "fix typo"
git commit -m "again"
# ... правим src/components/TaskItem.tsx ...
git commit -m "wip filter ui"
git commit -m "oops"
git commit -m "more"
```

**Граф ДО переписывания:**

```bash
git log --oneline --graph -8
```

```
* 65dea17 more
* 09abdd9 oops
* 7dcadb3 wip filter ui
* 54282b6 again
* a1d1d3f fix typo
* 2243bfd wip
* 2381b21 docs: add module 0 walkthrough (objects, three trees, first commit)
* 6061d7c chore: initial taskflow scaffold (Vite + React + TS)
```

**Объяснение:** шесть коммитов сверху — это «мусорная» история одной фичи. Первая
тройка (`wip`/`fix typo`/`again`) — это итерации по `taskUtils.ts`, вторая
(`wip filter ui`/`oops`/`more`) — по `TaskItem.tsx`. По таким сообщениям через
неделю невозможно понять, что произошло. Перед ревью это надо причесать.

**Публикуем черновую ветку (открываем PR).** В реальной работе на этом этапе ветку уже
выкладывают на сервер — показать прогресс, прогнать CI, открыть черновой Pull Request:

```bash
git push -u origin feature/task-search
```

```
 * [new branch]      feature/task-search -> feature/task-search
branch 'feature/task-search' set up to track 'origin/feature/task-search'.
```

Теперь на сервере (вашем GitHub из модуля 5) лежит именно эта **«грязная»** история.
Дальше мы причешем её локально (шаги 1–2) — и локальная версия разойдётся с серверной.
Вот тогда и понадобится **force-push** (шаг 3). Если бы мы чистили историю **до** первого
push — форс не потребовался бы вовсе.

## Шаг 1. Interactive rebase: складываем 6 коммитов в 2

Запускаем интерактивный rebase по последним 6 коммитам:

```bash
git rebase -i HEAD~6
```

Открывается todo-список (сверху — самый старый коммит). Приводим его к такому виду:

```
pick   2243bfd wip
squash a1d1d3f fix typo
squash 54282b6 again
reword 7dcadb3 wip filter ui
fixup  09abdd9 oops
fixup  65dea17 more
```

**Что мы здесь сказали Git:**

- первую тройку (`taskUtils.ts`) слить в один коммит через `squash` — Git предложит
  объединённое сообщение, которое мы перепишем начисто;
- второй коммит группы UI (по `TaskItem.tsx`) оставить базовым, но через `reword` дать
  ему нормальное имя, а две его доработки (`oops`, `more`) подклеить через `fixup` — их «мусорные»
  сообщения просто выбрасываются.

Сохраняем todo — Git проигрывает план. На шаге `squash` он открывает редактор с
объединённым черновиком сообщения:

```
# This is a combination of 3 commits.
# This is the 1st commit message:

wip

# This is the commit message #2:

fix typo

# This is the commit message #3:

again
...
```

Заменяем весь этот черновик на одну осмысленную строку
`feat(utils): add searchTasks helper`, а для второго (ui) коммита на `reword` вводим
`feat(ui): highlight search match in TaskItem`.

**Реальный вывод rebase:**

```
[detached HEAD 8ea5e83] feat(ui): highlight search match in TaskItem
 Date: Sat Jun 20 10:00:00 2026 +0100
 1 file changed, 3 insertions(+)
Successfully rebased and updated refs/heads/feature/task-search.
```

**Граф ПОСЛЕ:**

```bash
git log --oneline --graph -4
```

```
* e3bbe34 feat(ui): highlight search match in TaskItem
* 458584a feat(utils): add searchTasks helper
* 2381b21 docs: add module 0 walkthrough (objects, three trees, first commit)
* 6061d7c chore: initial taskflow scaffold (Vite + React + TS)
```

**Объяснение построчно:**

- Было 6 — стало **2** осмысленных коммита. Хеши **полностью новые** (`458584a`,
  `e3bbe34`): это копии, а не исходные коммиты.
- `detached HEAD` в выводе — нормальный режим работы rebase: Git временно
  перепроигрывает коммиты «вне ветки», а в конце переставляет указатель
  `feature/task-search` на верхушку.
- Главное — **итоговое содержимое файлов сохранено целиком**. Проверим:

```bash
git show HEAD~1:src/utils/taskUtils.ts | grep -A4 'export function searchTasks'
```

```
export function searchTasks(tasks: Task[], query: string): Task[] {
  const q = query.trim().toLowerCase()
  if (!q) return tasks
  return tasks.filter((t) => t.title.toLowerCase().includes(q))
}
```

В коммите видна финальная версия со всеми тремя итерациями — squash слил их содержимое,
а не выбросил.

> 💡 Чтобы **переупорядочить** коммиты, достаточно поменять местами строки в todo.
> Чтобы **выкинуть** коммит — заменить `pick` на `drop` (или удалить строку). Эти
> операции отрабатываем в домашнем задании.

## Шаг 2. «Правильный» способ: `--fixup` + `--autosquash`

Складывать руками в todo — нормально, но есть более аккуратный приём для типичного
случая «нашёл баг в уже сделанном коммите и хочу подшить правку именно к нему».

Допустим, ветка уже причёсана (2 коммита). Мы дописали в `taskUtils.ts` функцию
`countMatches` — это доработка коммита `458584a`. Вместо нового «fix»-коммита делаем
**fixup, нацеленный на конкретный коммит**:

```bash
git add src/utils/taskUtils.ts
git commit --fixup 458584a
```

**Вывод:**

```
[feature/task-search 5ecaf55] fixup! feat(utils): add searchTasks helper
 1 file changed, 4 insertions(+)
```

**Объяснение:** Git сам сделал сообщение `fixup! <заголовок целевого коммита>`. Это
**метка**: «меня надо подклеить к тому коммиту». Сейчас в логе временно три коммита:

```bash
git log --oneline -3
```

```
5ecaf55 fixup! feat(utils): add searchTasks helper
e3bbe34 feat(ui): highlight search match in TaskItem
458584a feat(utils): add searchTasks helper
```

Теперь запускаем rebase с `--autosquash` — Git **сам** найдёт `fixup!`-коммит и
поставит его в todo сразу после цели с командой `fixup`:

```bash
git rebase -i --autosquash 2381b21
```

todo, который Git **сгенерировал автоматически** (нам не пришлось ничего переставлять):

```
pick 458584a feat(utils): add searchTasks helper
fixup 5ecaf55 fixup! feat(utils): add searchTasks helper
pick e3bbe34 feat(ui): highlight search match in TaskItem
```

Сохраняем — Git доигрывает:

```
Successfully rebased and updated refs/heads/feature/task-search.
```

**Результат:**

```bash
git log --oneline -3
```

```
1a8a637 feat(ui): highlight search match in TaskItem
d120f28 feat(utils): add searchTasks helper
2381b21 docs: add module 0 walkthrough (objects, three trees, first commit)
```

```bash
git show HEAD~1:src/utils/taskUtils.ts | grep countMatches
```

```
export function countMatches(tasks: Task[], query: string): number {
```

**Объяснение:** `fixup!`-коммит **исчез** — его изменения вшиты в `d120f28`
(сообщение целевого коммита сохранилось, ничего лишнего в истории нет). Хеши снова
новые. Это самый чистый рабочий цикл: пока пилите фичу, на каждую доработку делаете
`git commit --fixup <hash>`, а перед ревью один раз `git rebase --autosquash` всё
схлопывает по местам.

> 💡 Чтобы не писать `--autosquash` каждый раз, включите его по умолчанию:
> `git config --global rebase.autosquash true`.

## Шаг 3. Безопасный force-push в свою PR-ветку

Ветку мы опубликовали ещё в шаге 0 — на сервере лежит её **старая, «грязная»** версия
(6 коммитов). После переписывания (шаги 1–2) локальная история стала другой, и две
версии ветки разошлись: у сервера — старая цепочка, у нас — новая.

```bash
git push origin feature/task-search
```

**Вывод — обычный push отклонён:**

```
 ! [rejected]        feature/task-search -> feature/task-search (non-fast-forward)
error: failed to push some refs to '...'
hint: Updates were rejected because the tip of your current branch is behind
hint: its remote counterpart. If you want to integrate the remote changes,
hint: use 'git pull' before pushing again.
```

**Объяснение:** Git защищает удалённую ветку: ваша верхушка — это не «продолжение»
серверной, а **параллельная** история. Делать `git pull` тут НЕ нужно (он притащит
обратно старые коммиты и устроит лишний merge). Нужно осознанно перезаписать ветку:

```bash
git push --force-with-lease origin feature/task-search
```

**Вывод:**

```
 + 1a8a637...fafa0cd feature/task-search -> feature/task-search (forced update)
```

**Объяснение:** `+` и `forced update` — ветка на сервере перезаписана новой историей.
`--force-with-lease` сделал это **только потому, что** удалённая ветка была там, где мы
её в последний раз видели. Если бы коллега успел запушить свой коммит — Git отказал бы
с `stale info`, и мы бы ничего не затёрли. В этом вся разница с «голым» `--force`.

> 🧹 `feature/task-search` была учебной — после экспериментов её можно убрать, чтобы не
> копить мусор: `git switch main`, затем `git branch -D feature/task-search` и
> `git push origin --delete feature/task-search`. `main` мы за весь модуль не трогали.

## Типичные ошибки модуля 7

- ❌ **Rebase общей ветки.** `git rebase` для `main`/`develop`/релизной ветки ломает
  историю всей команде: у коллег появляются дубликаты коммитов и каскад конфликтов.
  Переписываем только своё личное.
- ❌ **`git push -f` / `--force` вместо `--force-with-lease`.** `-f` — это короткая форма
  «голого» `--force`; он молча затирает чужие коммиты, прилетевшие в ветку, пока вы
  переписывали историю. Привыкайте к `--force-with-lease` как к рефлексу.
- ❌ **Делать `git pull` после rebase «чтобы помириться».** Это тащит обратно старые
  коммиты и плодит мусорный merge. После rebase своей ветки — именно
  `push --force-with-lease`, а не `pull`.
- ❌ **Страх «потерять коммиты».** Они никуда не пропадают сразу: лежат в **reflog** и
  достижимы по хешу. Восстановление — модуль 8.
- ❌ **Удалить строку в todo «случайно».** Удалённая строка = `drop` = коммит выпадет
  из истории. Если очистить todo полностью — rebase отменится (это аварийный выход).

## Шпаргалка команд модуля 7

```bash
# линеаризация: перенести свою ветку поверх ушедшего вперёд main
git rebase main                     # переиграть свои коммиты от верхушки main

# интерактивное переписывание последних N коммитов
git rebase -i HEAD~5                # открыть todo: pick/reword/edit/squash/fixup/drop
                                    #   + переставить строки = reorder

# подшить правку к конкретному прошлому коммиту (правильный цикл)
git commit --fixup <hash>           # создать fixup!-метку на коммит <hash>
git rebase -i --autosquash main     # авто-расставить fixup!/squash! и схлопнуть
git config --global rebase.autosquash true   # включить --autosquash по умолчанию

# управление прерванным rebase
git rebase --continue               # продолжить после edit / разрешения конфликта
git rebase --abort                  # откатить всё к состоянию до rebase
git rebase --edit-todo              # пересмотреть оставшийся план

# безопасный force-push своей PR-ветки
git push --force-with-lease origin <branch>   # форс, только если сервер там, где ждём
# git push --force  (то же: git push -f)       # ОПАСНО: затирает чужое молча — избегать
```

### Часть A. Линеаризация через `git rebase main`

1. Сделайте в ветке `feature/sort-tasks` **2 коммита** (например, мелкая правка в
   `src/utils/taskUtils.ts` и в `tests/taskUtils.test.ts`).
2. Переключитесь на `main` и сделайте там **1 коммит**, чтобы `main` ушёл вперёд
   (например, правка в `README.md`). Теперь ветки **разошлись**.
3. Посмотрите расходящийся граф: `git log --oneline --graph --all`.
4. Вернитесь в ветку и выполните `git rebase main`.

**Критерии «сделано»:**

- `git log --oneline --graph --all` показывает **линейную** историю: ваши 2 коммита
  стоят **поверх** коммита из `main`, без «вилки» и без merge-коммита.
- Хеши ваших двух коммитов **изменились** относительно шага 3 (это копии).

**Подсказка:** если при rebase возникнет конфликт — это нормально; правьте файл,
`git add <файл>`, затем `git rebase --continue`. Передумали — `git rebase --abort`.

### Часть B. Причёсывание: reorder + reword + drop + edit

Сделайте в ветке **7 хаотичных коммитов** и приведите их к **3 чистым**, не потеряв
ни одного нужного изменения. Используйте `git rebase -i HEAD~7` и операции:

1. **reorder** — переставьте строки в todo так, чтобы связанные коммиты шли рядом
   (например, все правки по `sortTasks` подряд).
2. **reword** — дайте двум коммитам осмысленные сообщения в стиле `feat(...)`/`fix(...)`
   вместо `wip`/`tmp`.
3. **drop** — выкиньте один заведомо лишний коммит (например, отладочный
   `console.log`), заменив `pick` на `drop`.
4. **edit** — остановитесь на **одном коммите посередине**, поправьте в нём файл
   (`git add ...; git commit --amend --no-edit`) и продолжите `git rebase --continue`.

**Критерии «сделано»:**

- В итоге в ветке ровно **3** осмысленных коммита поверх базы.
- Изменения из «лишнего» коммита (drop) в финальных файлах **отсутствуют**, а из всех
  остальных — **присутствуют** (проверьте `git show` и содержимое файлов).
- У переименованных коммитов новые сообщения; порядок соответствует задуманному.
- `npm test` (или `npx vitest run`) проходит — переписывание не сломало код.

**Подсказки:**

- Во время `edit` смотрите `git status` — Git подскажет, что вы «editing a commit
  while rebasing», и предложит `--amend` / `--continue`.
- Сверяйтесь с «Шпаргалкой команд» выше: `--continue`, `--abort`, `--edit-todo`.
- Не паникуйте, если что-то пошло не так: `git rebase --abort` вернёт всё назад.

### Часть C. Безопасный force-push

1. Запушьте ветку на свой `origin` (GitHub из модуля 5) **до** причёсывания — это
   имитирует «PR уже открыт»: `git push -u origin feature/sort-tasks`.
2. После переписывания истории попробуйте обычный `git push` — убедитесь, что он
   **отклонён** (non-fast-forward).
3. Выполните `git push --force-with-lease origin feature/sort-tasks`.

**Критерии «сделано»:**

- Обычный push был отклонён с `non-fast-forward`.
- `--force-with-lease` прошёл: вывод содержит `forced update`, а локальная и удалённая
  верхушки совпадают (`git log -1` и `git log -1 origin/feature/sort-tasks`).
