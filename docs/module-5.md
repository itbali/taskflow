# Модуль 5. Создание удалёнки и клонирование

> **Цель модуля.** Связать локальный репозиторий с удалённым: создать репозиторий на
> GitHub, подключить его через `git remote add`, опубликовать историю первым
> `git push -u`. Разобраться, как **получают** код: `git clone` по **HTTPS** и **SSH**,
> скачивание **ZIP**, и в чём разница между **clone** и **fork**. Это «онбординг на
> GitHub» — дальше, в модуле 6, начнётся ежедневная синхронизация (`fetch`/`pull`/PR).

До этого момента (модули 0–4) мы работали **полностью локально**: вся история жила в
`.git/` на диске, и никакой сервер был не нужен. Теперь подключим к проекту `taskflow`
удалённую копию — чтобы хранить бэкап, работать с другими и подключить CI.

---

## Мысленная модель (прочитать до команд)

Вспомните идею из модуля 0: **Git распределённый**, у вас на диске лежит полный
репозиторий целиком. **Удалённый репозиторий (remote)** — это просто **ещё одна копия**
того же репозитория на сервере. Он не «главнее» вашего — он равноправный, вы лишь
**синхронизируете** копии между собой.

```
ваш ноутбук (.git/)            GitHub (origin)
  main ●──●──●        ⇄          main ●──●──●
        локально              «ещё одна копия» того же репо
```

- **`origin`** — это просто **имя по умолчанию** для удалёнки (можно было назвать как
  угодно). За именем прячется URL сервера.
- **Зачем remote:** бэкап истории; совместная работа (другие видят ваши коммиты);
  площадка для PR, ревью, CI/CD; публикация релизов.
- **Важно:** создать удалёнку и запушить — это **не** «переехать в облако». Локальный
  репозиторий остаётся полноценным; remote — синхронизируемая копия.

---

## Шаг 1. Создаём пустой репозиторий на GitHub

Через веб-интерфейс: **New repository** → имя `taskflow` → **не** добавляйте README,
`.gitignore` и лицензию (репозиторий должен быть **пустым**, иначе истории разойдутся
при первом push). GitHub покажет подсказку «…or push an existing repository».

Через `gh` CLI (официальный GitHub CLI) — то же самое одной командой:

```bash
gh repo create taskflow --private --source=. --remote=origin
```

- `--source=.` — взять текущий локальный репозиторий;
- `--remote=origin` — сразу прописать его как `origin`;
- `--private` / `--public` — видимость.

> 💡 Если используете `gh repo create --source=.`, шаги 2 ниже (`remote add` + первый
> `push`) делаются автоматически. Разберём их вручную — чтобы понимать, что происходит.

---

## Шаг 2. Подключаем удалёнку и публикуем историю

Связываем локальный репозиторий с пустым удалённым и отправляем все коммиты:

```bash
git remote add origin git@github.com:<user>/taskflow.git
git remote -v
```

**Вывод `git remote -v`:**

```
origin  git@github.com:<user>/taskflow.git (fetch)
origin  git@github.com:<user>/taskflow.git (push)
```

- `git remote add <имя> <url>` — заводит удалёнку под именем `origin`. Это пока только
  запись в `.git/config`, ничего ещё не отправлено.
- Две строки (`fetch`/`push`) — один и тот же URL используется и для скачивания, и для
  отправки (можно развести, но обычно не нужно).

Отправляем историю первым push:

```bash
git push -u origin main
```

**Вывод:**

```
Enumerating objects: 24, done.
...
To github.com:<user>/taskflow.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.
```

- `-u` (`--set-upstream`) — **привязывает** локальную `main` к `origin/main`. После этого
  достаточно `git push`/`git pull` без аргументов — Git знает, куда. Эта **upstream-связь**
  и есть `branch 'main' set up to track 'origin/main'`.
- `* [new branch] main -> main` — на сервере создана ветка `main` с вашими коммитами.
- Теперь в `git log --decorate` рядом с `HEAD -> main` появится и `origin/main` —
  указатель на состояние ветки **на сервере** (каким его видит ваш локальный Git).

### Публикация тегов

Важный нюанс: `git push` сам по себе **теги не отправляет** — даже если они указывают на
уже запушенные коммиты. Теги (мы создавали их в модуле 4) публикуют **отдельно**:

```bash
git push origin v0.1.0        # отправить один конкретный тег
git push origin --tags        # отправить ВСЕ локальные теги
git push --follow-tags        # коммиты + аннотированные теги, на них указывающие
```

- `--tags` шлёт **все** теги, включая случайные/временные — пользоваться аккуратно.
- `--follow-tags` — рекомендуемый вариант для релизов: уходят только аннотированные теги,
  привязанные к пушащимся коммитам.
- Удалить тег на удалёнке: `git push origin --delete v0.1.0`.

На GitHub запушенный тег автоматически появляется в разделе **Releases** — на его основе
оформляют release с описанием и собранными артефактами.

## Шаг 3. Как получают код: clone (HTTPS / SSH) и ZIP

Если репозиторий уже есть на сервере, его **клонируют** — скачивают вместе со всей
историей и сразу с настроенным `origin`.

### HTTPS

```bash
git clone https://github.com/<user>/taskflow.git
```

- Работает везде, не требует настройки ключей.
- Аутентификация для приватных репо и для push — по **Personal Access Token** (не по
  паролю аккаунта; пароль для Git GitHub давно отключил).

### SSH

```bash
git clone git@github.com:<user>/taskflow.git
```

- Аутентификация по **SSH-ключу** — один раз настроил, дальше push/pull без ввода
  токена. Предпочтительно для повседневной работы.
- Разовая настройка ключа:

```bash
ssh-keygen -t ed25519 -C "you@example.com"   # создать ключ (Enter на все вопросы)
cat ~/.ssh/id_ed25519.pub                     # СКОПИРОВАТЬ публичный ключ
# вставить его в GitHub → Settings → SSH and GPG keys → New SSH key
ssh -T git@github.com                          # проверка: "Hi <user>! You've successfully authenticated"
```

> **HTTPS vs SSH:** функционально одинаково. HTTPS проще начать (токен), SSH удобнее в
> долгую (ключ вместо токена). URL у них разный — отсюда и два варианта в кнопке «Code»
> на GitHub.

### ZIP — это НЕ клонирование

Кнопка **Code → Download ZIP** скачивает только **текущий снимок файлов** — **без**
папки `.git/`, то есть **без истории и без связи с удалёнкой**. Это не репозиторий, а
просто архив кода. Подходит, когда нужно «просто посмотреть/собрать» и история не важна;
для работы с Git — всегда `clone`.

## Шаг 4. clone vs fork — когда что

| | `git clone` | **fork** |
| --- | --- | --- |
| Что делает | копию репозитория **себе на машину** | копию чужого репозитория **в ваш аккаунт на сервере** |
| Где живёт | локально | на GitHub (ваша копия) + потом клонируете её |
| Когда | свои репо или те, куда есть доступ на push | контрибуция в **чужой** проект, где прав на push нет |

Типичный сценарий контрибуции в чужой проект:

```
upstream (чужой репо)
   │ fork (на GitHub)
   ▼
origin (ваша копия на GitHub)
   │ git clone
   ▼
локально → коммиты → git push origin → Pull Request в upstream
```

- **fork** делается кнопкой на GitHub (или `gh repo fork`). Появляется ваша серверная
  копия.
- Её вы **клонируете**; при этом `origin` = ваш форк, а оригинал принято добавить
  вторым remote под именем **`upstream`** (`git remote add upstream <url-оригинала>`),
  чтобы подтягивать чужие обновления.
- Изменения уходят в `origin` (ваш форк), а в оригинал предлагаются через **Pull
  Request** (подробно — модуль 6).

## Что `clone` настраивает за вас

`git clone` — это не только «скачать файлы». Он автоматически:

- создаёт папку с `.git/` и **всей историей**;
- прописывает удалёнку `origin` на URL, откуда клонировали;
- выкачивает все ветки сервера и создаёт локальную ветку по умолчанию (`main`),
  **привязанную** к `origin/main` (та самая upstream-связь из шага 2).

Поэтому после `clone` сразу работают `git push`/`git pull` без аргументов.

## Типичные ошибки модуля 5

- ❌ **Создать репозиторий на GitHub с README/`.gitignore`, а потом пушить локальный.**
  Истории разойдутся → `push` отклонят. Создавайте **пустой** репо (или потом
  `git pull --rebase`, но это уже модуль 6).
- ❌ **Скачать ZIP и ждать, что заработает Git.** В ZIP нет `.git/` — это не репозиторий.
- ❌ **Путать clone и fork.** Нет прав на push в чужой репо → нужен fork, а не clone.
- ❌ **Пытаться пушить по паролю аккаунта.** GitHub принимает токен (HTTPS) или
  SSH-ключ, не пароль.
- ❌ **Забыть `-u` при первом push** → потом приходится каждый раз указывать
  `git push origin main` вручную.

## Шпаргалка команд модуля 5

```bash
# создать удалёнку и опубликовать локальный репозиторий
gh repo create taskflow --private --source=. --remote=origin   # через GitHub CLI
git remote add origin <url>          # подключить вручную
git remote -v                        # посмотреть удалёнки
git push -u origin main              # первый push + upstream-связь
git push --follow-tags               # отправить коммиты + аннотированные теги

# получить чужой/свой репозиторий
git clone https://github.com/<user>/<repo>.git   # по HTTPS (токен)
git clone git@github.com:<user>/<repo>.git       # по SSH (ключ)

# настройка SSH-ключа (один раз)
ssh-keygen -t ed25519 -C "you@example.com"
cat ~/.ssh/id_ed25519.pub            # добавить в GitHub → SSH keys
ssh -T git@github.com                # проверка

# работа с форком
gh repo fork <owner>/<repo>          # форкнуть чужой репозиторий
git remote add upstream <url-оригинала>   # подтягивать обновления оригинала
```

## Домашнее задание

Опубликуйте свой `taskflow` на GitHub и потренируйте способы клонирования.

### Что нужно сделать

1. Создайте на GitHub **пустой** репозиторий и подключите его как `origin`.
2. Опубликуйте историю первым `git push -u origin main`; убедитесь, что коммиты видны
   на GitHub.
3. Запушите теги из модуля 4: `git push origin --follow-tags`.
4. Склонируйте свой репозиторий в **другую папку** двумя способами — по HTTPS и по SSH
   (настроив ключ), и сравните, чем отличается процесс аутентификации.
5. Найдите любой публичный репозиторий, **форкните** его и склонируйте форк; добавьте
   оригинал как `upstream`.

### Критерии «сделано»

- [ ] `git remote -v` показывает `origin` с вашим URL.
- [ ] На GitHub видны все коммиты и теги.
- [ ] `git push`/`git pull` без аргументов работают (есть upstream-связь).
- [ ] Есть локальный клон по SSH, сделанный без ввода токена (ключ настроен).
- [ ] У клонированного форка `git remote -v` показывает и `origin` (форк), и `upstream`.

### Подсказки

- Если при первом push видите «Updates were rejected» — значит, репо на GitHub был
  создан **не пустым**. Либо пересоздайте пустым, либо `git pull --rebase origin main`
  (механику разберём в модуле 6).
- Проверить, по HTTPS или SSH настроен `origin`: `git remote -v` — URL начинается с
  `https://` или с `git@`.
