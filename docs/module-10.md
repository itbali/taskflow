# Модуль 10. Хирургия по истории: revert, cherry-pick, bisect

> **Цель модуля.** Освоить три точечных инструмента работы с уже существующей
> историей: безопасный откат **опубликованного** коммита (`revert`), перенос **одного**
> коммита между ветками (`cherry-pick`) и автоматический поиск регрессии (`bisect`). К
> концу модуля вы своими словами объясняете разницу `revert` и `reset` (модуль 8) и
> умеете находить «плохой» коммит через `git bisect run` без ручного перебора.

Этот документ — пошаговый разбор «живого примера» модуля 10 на проекте `taskflow`. Все
выводы команд ниже — **настоящие**. Единственное, что у вас будет отличаться, — сами
хеши (SHA-1) и дата/автор в служебных строках.

---

## Мысленная модель (прочитать до команд)

Три идеи, на которых держится весь модуль.

1. **revert создаёт новый коммит, а не переписывает историю.**

   ```
   reset  → переписывает, КУДА смотрит ветка (история «переезжает» назад, модуль 8)
   revert → добавляет НОВЫЙ коммит, отменяющий изменения старого (история растёт вперёд)
   ```

   `revert` ничего не переписывает, только добавляет «обратный» коммит → безопасен для
   уже опубликованного: это штатный способ откатить баг в общей ветке, не ломая историю
   коллегам. Если бы коммит был **локальным** и ещё не отправлен — можно было бы стереть
   его `git reset --hard HEAD~1` (модуль 8) без следов, но для **опубликованного**
   правильный инструмент — именно `revert`.

2. **cherry-pick копирует изменение, а не сам коммит.** Коммит применяется поверх
   текущей ветки как **новый** объект — с тем же диффом и сообщением, но с другим
   родителем, а значит и другим хешем. Это отличает его от `merge`/`rebase`, которые
   переносят **диапазон** коммитов целиком.

3. **bisect = двоичный поиск по истории.** Вместо ручного перебора «когда же сломалось»
   вы помечаете один заведомо «хороший» и один «плохой» коммит, а Git делит диапазон
   пополам. С `git bisect run <команда>` Git сам прогоняет проверку на каждом шаге:
   за `log2(N)` шагов находит **первый плохой коммит**.

## Шаг 1. revert опубликованного бага

Представим, что в `main` уехал баг — и его уже видели коллеги. Переписывать историю
(`reset`) нельзя — откатываем штатно, через `revert`.

```bash
# правка вносит баг в countRemaining
git add src/utils/taskUtils.ts && git commit -m "feat(utils): tweak countRemaining rounding"
git log --oneline -2
```

**Вывод:**

```
967e2d9 feat(utils): tweak countRemaining rounding
b71ece7 chore: add .gitattributes for line-ending normalization
```

Откатываем баг, **не трогая историю**:

```bash
git revert --no-edit HEAD
git log --oneline -3
```

**Вывод:**

```
[main 91972ed] Revert "feat(utils): tweak countRemaining rounding"
 Date: Sat Jul 4 19:32:39 2026 +0100
 1 file changed, 1 insertion(+), 1 deletion(-)
91972ed Revert "feat(utils): tweak countRemaining rounding"
967e2d9 feat(utils): tweak countRemaining rounding
b71ece7 chore: add .gitattributes for line-ending normalization
```

**Объяснение:**
- Баговый коммит `967e2d9` **остался** в истории — он опубликован, мы его не прячем.
- Сверху появился **новый** коммит `91972ed`, который ровно отменяет его изменения
  (`countRemaining` снова считает правильно). Историю можно безопасно пушить — у всех
  она сойдётся.
- `--no-edit` — взять стандартное сообщение `Revert "..."` без открытия редактора.

## Шаг 2. cherry-pick одного коммита в релизную ветку

Фича `feature/lookup` ещё не готова целиком, но один коммит из неё нужен прямо в релиз.
Берём точечно через `cherry-pick`.

```bash
git checkout -b feature/lookup
cat >> src/utils/taskUtils.ts <<'EOF'

/** Найти задачу по id. */
export function getTaskById(tasks: Task[], id: string): Task | undefined {
  return tasks.find((t) => t.id === id)
}
EOF
git add src/utils/taskUtils.ts && git commit -m "feat: add getTaskById helper"
git rev-parse --short HEAD
```

**Вывод:** `6098e64`

```bash
git checkout -b release/1.0 main
git cherry-pick 6098e64
git log --oneline -3
```

**Вывод:**

```
[release/1.0 682c759] feat: add getTaskById helper
 Date: Sat Jul 4 19:32:47 2026 +0100
 1 file changed, 5 insertions(+)
682c759 feat: add getTaskById helper
91972ed Revert "feat(utils): tweak countRemaining rounding"
967e2d9 feat(utils): tweak countRemaining rounding
```

**Объяснение:**
- Изменения коммита `6098e64` применились поверх `release/1.0` как **новый** коммит
  `682c759`. Содержимое то же, но **хеш другой** — у него другой родитель (другое место
  в графе). cherry-pick копирует изменение, а не сам объект.
- Остальные коммиты `feature/lookup` не приехали — взяли ровно один, что и требовалось.

> 💡 `cherry-pick` переносит коммит **внутри одного репозитория**. Если общей объектной
> базы нет (другой репозиторий, нет общего remote), тот же перенос делают через
> патч-файлы — `git format-patch` / `git am`, см. модуль 11.

## Шаг 3. bisect: ловим спрятанную регрессию автоматически

Ключевая сцена модуля. В `taskUtils.ts` за несколько коммитов спрятан баг, который
ломает тест `sortTasks`. Искать вручную долго — отдадим поиск Git.

### 3.1. Подготовка серии коммитов

Чтобы `git bisect run npm test` мог прогонять тесты, зависимости уже установлены
(`npm ci` один раз при старте курса). Собираем ветку из 9 коммитов от актуального
`main`. Большинство — безобидные заметки, а **четвёртый** вносит регрессию в
`sortTasks` (меняет знак сравнения по `done`, из-за чего выполненные всплывают наверх):

```bash
git checkout -b dev/sprint main          # GOOD-база: 91972ed
# 3 безобидных коммита ...
# 4-й — РЕГРЕССИЯ в src/utils/taskUtils.ts:
#   было:  if (a.done !== b.done) return a.done ? 1 : -1
#   стало: if (a.done !== b.done) return a.done ? -1 : 1
git commit -m "refactor: tweak sortTasks ordering"   # b35131f — здесь баг
# ещё 5 безобидных коммитов сверху ...
git rev-list --count main..HEAD
```

**Вывод:**

```
9
```

Проверяем, что на вершине ветки тест действительно красный:

```bash
npm test
```

**Вывод (фрагмент):**

```
 ❯ tests/taskUtils.test.ts (6 tests | 1 failed) 14ms
   × sortTasks > ставит невыполненные выше и сортирует по createdAt
     → expected [ 'b', 'a', 'c' ] to deeply equal [ 'a', 'c', 'b' ]
 Test Files  1 failed (1)
      Tests  1 failed | 5 passed (6)
```

### 3.2. Запуск двоичного поиска

```bash
git bisect start
git bisect bad  c89331a      # вершина — точно сломана
git bisect good 91972ed      # база main — точно работала
```

**Вывод:**

```
status: waiting for both good and bad commits
status: waiting for good commit(s), bad commit known
Bisecting: 4 revisions left to test after this (roughly 2 steps)
[b35131fa234a8652e692a9fecf7b79253e50ac14] refactor: tweak sortTasks ordering
```

Git уже сообщил: осталось \~2 шага на 9 коммитов (это и есть `log2`). Теперь — самое
мощное: пусть Git сам прогонит тесты на каждом шаге.

```bash
git bisect run npm test
```

**Вывод (сокращён — оставлены вердикты Git между прогонами):**

```
running 'npm' 'test'
   × sortTasks > ставит невыполненные выше и сортирует по createdAt
     → expected [ 'b', 'a', 'c' ] to deeply equal [ 'a', 'c', 'b' ]
 Test Files  1 failed (1)
Bisecting: 1 revision left to test after this (roughly 1 step)
[2d0349e1adc81876cd209e12df61cbaea635a7b5] chore: sprint note 2
running 'npm' 'test'
 ✓ tests/taskUtils.test.ts (6 tests) 5ms
Bisecting: 0 revisions left to test after this (roughly 0 steps)
[8594aa503f28626295aafce64d2aedac9a283c02] chore: sprint note 3
running 'npm' 'test'
 ✓ tests/taskUtils.test.ts (6 tests) 4ms

b35131fa234a8652e692a9fecf7b79253e50ac14 is the first bad commit
commit b35131fa234a8652e692a9fecf7b79253e50ac14
Author: itbali <xopycaku@gmail.com>
    refactor: tweak sortTasks ordering
 src/utils/taskUtils.ts | 2 +-
 1 file changed, 1 insertion(+), 1 deletion(-)
bisect found first bad commit
```

**Объяснение, что произошло:**
- Git использовал **код возврата** `npm test`: `0` = good, не-`0` = bad. Поэтому
  `git bisect run` работает с любым скриптом, возвращающим осмысленный exit code.
- На каждом шаге диапазон делился пополам: `9 → 4 → 1 → 0`. Всего 3 прогона вместо
  перебора 9 коммитов вручную.
- Итог: `b35131f … is the first bad commit` — Git точно указал **тот самый** коммит с
  регрессией и показал его дифф (`taskUtils.ts | 2 +-`).

### 3.3. Завершение

```bash
git bisect reset
```

**Вывод:**

```
Previous HEAD position was 8594aa5 chore: sprint note 3
Switched to branch 'dev/sprint'
```

`git bisect reset` возвращает `HEAD` туда, где вы были до поиска. Дальше — чините
найденный коммит обычным образом (`revert` или fix-коммит).

> 💡 Если какой-то коммит невозможно протестировать (не собирается по другой причине),
> в ручном bisect используйте `git bisect skip`, а в `run`-скрипте — код возврата
> `125`: Git поймёт, что этот коммит надо пропустить.

## Типичные ошибки модуля 10

- ❌ **revert там, где нужен reset (и наоборот).** Для **локального** мусорного коммита
  — `reset` (модуль 8); для **опубликованного** бага — `revert`. Перепутать = либо
  мусор в истории, либо сломанная история у коллег.
- ❌ **Ручной поиск регрессии** перебором коммитов вместо `git bisect run`. На длинной
  истории это часы вместо `log2(N)` автоматических прогонов.
- ❌ **Забыть `git bisect reset`** после поиска — останетесь на «detached HEAD» в
  середине истории и запутаетесь.
- ❌ **Считать, что cherry-pick сохраняет хеш.** Хеш **всегда** новый — родитель другой,
  даже если содержимое коммита идентично.

## Шпаргалка команд модуля 10

```bash
# откат без переписывания истории
git revert <commit>              # новый «обратный» коммит (для опубликованного)
git revert --no-edit <commit>    # то же, со стандартным сообщением
git revert -m 1 <merge-commit>   # откатить MERGE-коммит (см. ДЗ)

# перенос отдельного коммита
git cherry-pick <commit>         # скопировать изменение в текущую ветку

# двоичный поиск регрессии
git bisect start
git bisect bad  <ref>            # заведомо сломанный коммит
git bisect good <ref>            # заведомо рабочий коммит
git bisect run npm test          # автопоиск по коду возврата (0=good, 1..124=bad, 125=skip)
git bisect reset                 # выйти из режима bisect
```

## Домашнее задание

Живые примеры выше уже показали: `revert` бага, `cherry-pick`, `bisect` по сломанному
`sortTasks`. В ДЗ — те же навыки, но другие сценарии. Работайте в своём `taskflow`, в
**отдельных учебных ветках** от `main` (`hw/...`).

### Задание 1. Откатить MERGE-коммит через `git revert -m 1`

В живом примере мы реверти­ли **обычный** коммит. Merge-коммит реверти­тся иначе — у
него два родителя, и Git нужно подсказать, какую сторону считать «основной».

**Шаги:**
1. От `main` создайте ветку `hw/feature`, сделайте на ней 1–2 коммита (правка любого
   файла, например добавьте функцию-заглушку в `src/utils/taskUtils.ts`).
2. Вернитесь на `main` и слейте ветку **без fast-forward**: `git merge --no-ff hw/feature`.
   Убедитесь, что появился отдельный merge-коммит (`git log --graph --oneline`).
3. Решите, что фичу нужно откатить целиком. Выполните `git revert -m 1 <merge-commit>`.
4. Проверьте, что изменения фичи отменены, а история (включая merge) сохранена.

**Критерии «сделано»:**
- В истории есть merge-коммит с двумя родителями (`git show <merge> | head`).
- После `revert -m 1` файлы вернулись к состоянию до слияния.
- Добавлен новый revert-коммит; merge-коммит **не** исчез.

**Подсказки:** `-m 1` означает «оставить первого родителя (mainline) как базу и
отменить то, что принесла вторая сторона». Без `-m` Git откажется реверти­ть merge с
ошибкой `commit is a merge but no -m option was given`.

**Самопроверка:** `git diff <состояние-до-merge> HEAD -- <изменённый-файл>` должен
показать пустой дифф по содержимому фичи.

### Задание 2. cherry-pick с конфликтом

В живом примере cherry-pick прошёл чисто. На практике коммит часто конфликтует с
целевой веткой — научитесь это разруливать.

**Шаги:**
1. От `main` создайте `hw/base`, измените строку в `sortTasks` (`taskUtils.ts`) и
   закоммитьте.
2. Создайте `hw/feature` от того же `main` (**до** правки из шага 1), измените **ту же
   строку** по-другому и закоммитьте.
3. Переключитесь на `hw/base` и выполните `git cherry-pick <hash-коммита-из-hw/feature>`
   — должен возникнуть конфликт.
4. Разрешите конфликт руками, `git add`, затем `git cherry-pick --continue`.

**Критерии «сделано»:**
- В выводе `cherry-pick` видно сообщение о конфликте и маркеры `<<<<<<<`/`>>>>>>>` в
  файле.
- После `--continue` создан новый коммит, `git status` чист.

**Подсказки:** если запутались — `git cherry-pick --abort` откатывает попытку
полностью, как будто её и не было.

**Самопроверка:** `git log --oneline -3 hw/base` показывает перенесённый коммит с
новым хешем поверх исходных.

### Задание 3. bisect ДРУГОГО бага (регрессия в `countRemaining`)

В живом примере bisect ловил баг в `sortTasks`. Теперь спрячьте регрессию в
**`countRemaining`** и найдите её своим тестом.

**Шаги:**
1. Допишите в `tests/taskUtils.test.ts` тест, жёстко фиксирующий поведение
   `countRemaining` (например, что для смешанного списка он возвращает число
   невыполненных, а **пустой** список даёт `0`). Закоммитьте тест — это «good».
2. Создайте ветку и серию из \~10 коммитов; в одном из средних внесите регрессию в
   `countRemaining` (например, поменяйте условие на противоположное — считать
   выполненные вместо невыполненных).
3. На вершине убедитесь, что `npm test` красный.
4. Запустите `git bisect start`, отметьте `good`/`bad` и выполните
   `git bisect run npm test`. Зафиксируйте, какой коммит назван первым плохим.
5. Завершите `git bisect reset` и почините баг (`revert` или fix-коммит).

**Критерии «сделано»:**
- Новый тест на `countRemaining` падает ровно на коммите с регрессией.
- `git bisect run npm test` выводит `<hash> is the first bad commit`, и это
  действительно ваш коммит с багом.
- После `git bisect reset` вы вернулись на вершину ветки.

**Подсказки:** тест должен быть «детерминированным» — падать только из-за вашей
регрессии, иначе bisect укажет не туда. Если коммит не собирается по другой причине,
помечайте его кодом возврата `125` (skip), а не `1` (bad).

**Самопроверка:** сравните хеш из вывода bisect с хешем коммита, в котором вы вносили
регрессию (`git log --oneline` своей ветки) — они должны совпасть.
