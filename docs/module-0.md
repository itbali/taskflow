# Модуль 0. Основания и мысленная модель

> **Цель модуля.** Понять, _что именно_ Git хранит и почему. Без этой модели все
> дальнейшие команды заучиваются как заклинания. К концу модуля вы своими словами
> объясняете разницу **working tree / index / HEAD** и знаете, что такое
> **blob / tree / commit**.

Этот документ — пошаговый разбор «живого примера» модуля 0 на проекте `taskflow`.
Все выводы команд ниже — **настоящие**, снятые при инициализации этого репозитория.
Единственное, что у вас будет отличаться, — это сами хеши (SHA-1): они зависят от
содержимого, вашего имени/почты и времени коммита. Структура и смысл вывода — те же.

---

## Мысленная модель (прочитать до команд)

Три ключевые идеи, на которых держится весь Git:

1. **Git хранит снимки (snapshots), а не диффы.** Каждый коммит — это полный слепок
   состояния проекта на момент коммита. Диффы («что изменилось») Git вычисляет на
   лету, когда вы просите `git diff` или `git show`. Это противоположно интуиции
   многих систем, и именно из этого следует, почему ветки и переключения такие дешёвые.

2. **Три дерева, через которые проходит любое изменение:**

   ```
   working tree   →   index (staging)   →   repository (HEAD)
   рабочая копия       «корзина» к           зафиксированная
   файлов на диске     следующему коммиту    история коммитов
        |  git add ──────────►  |  git commit ──────────►  |
   ```

   - **working tree** — обычные файлы на диске, которые вы редактируете.
   - **index / staging** — промежуточная область: что попадёт в _следующий_ коммит.
   - **HEAD / repository** — сами коммиты, уже зафиксированная история.

3. **Объектная модель.** Внутри `.git/objects` лежат объекты четырёх типов, каждый
   адресуется своим хешем:
   - **blob** — содержимое одного файла (без имени!);
   - **tree** — каталог: список имён → ссылки на blob'ы и вложенные tree;
   - **commit** — снимок (ссылка на корневой tree) + метаданные (автор, дата,
     сообщение) + ссылка(и) на родительский коммит;
   - **tag** — именованная аннотированная метка (разберём в модуле 4).

   **HEAD** — указатель на текущую ветку, ветка — подвижный указатель на коммит,
   коммит — узел в графе. Запомните цепочку: `HEAD → ветка → commit → tree → blob'ы`.

> **Git — распределённая система.** В отличие от централизованных (SVN), где история
> живёт на сервере, у вас на диске лежит **полный репозиторий целиком**: вся история,
> все версии, все ветки — внутри папки `.git/`. Это **локальный** репозиторий, и для
> работы (коммиты, ветки, просмотр истории, откаты) сервер вообще не нужен — всё
> происходит офлайн, на вашей машине.
>
> **Удалённый репозиторий** (remote, например на GitHub) — это просто **ещё одна копия**
> того же репозитория на сервере, с которой вы время от времени **синхронизируетесь**:
> отправляете свои коммиты (`push`) и забираете чужие (`pull`/`fetch`). Сервер не
> «главнее» вашей копии — он равноправный. Здесь, в модулях 0–4, мы работаем **полностью
> локально**, без всякого удалённого; синхронизацию с remote подробно разберём в модуле 6.

## Шаг 0. Предварительная настройка Git

Это делается один раз на машину. Если уже настроено — просто проверьте значения.

```bash
git --version
git config --global user.name "Ваше Имя"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
git config --global core.editor "code --wait"
git config --global core.autocrlf input
```

**Что делает каждая команда:**

| Команда                     | Зачем                                                                                                                                                                                              |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `git --version`             | Проверить, что Git установлен, и узнать версию (некоторые флаги зависят от версии).                                                                                                                |
| `user.name` / `user.email`  | Эти данные **вшиваются в каждый коммит** как автор и коммиттер. На GitHub по e-mail коммит привязывается к вашему аккаунту.                                                                        |
| `init.defaultBranch main`   | Имя ветки по умолчанию в новых репозиториях. Без него Git создаёт `master` и показывает предупреждение (см. ниже).                                                                                 |
| `core.editor "code --wait"` | Какой редактор открывать для сообщений коммитов и interactive rebase. `--wait` заставляет VS Code дождаться закрытия вкладки.                                                                      |
| `core.autocrlf input`       | Политика окончаний строк. `input` = «при коммите конвертировать CRLF→LF, при checkout не трогать». Для macOS/Linux это безопасно, для Windows — частый источник «фантомных» изменений всего файла. |

**Ожидаемый вывод** `git --version`:

```
git version 2.43.0
```

> 💡 У `git config` без вывода — это норма: команда молча записывает значение в
> `~/.gitconfig`. Проверить можно `git config --global --list` или
> `git config --global user.name` (вернёт записанное значение).

## Шаг 1. Инициализация репозитория

```bash
cd taskflow
git init
```

**Ожидаемый вывод** (если `init.defaultBranch` НЕ настроен):

```
hint: Using 'master' as the name for the initial branch. This default branch name
hint: is subject to change. To configure the initial branch name to use in all
hint: of your new repositories, which will suppress this warning, call:
hint:
hint:   git config --global init.defaultBranch <name>
hint: ...
Initialized empty Git repository in /.../taskflow/.git/
```

**Объяснение вывода:**

- Блок `hint:` — не ошибка, а подсказка: имя стартовой ветки сейчас `master`, и Git
  предлагает задать дефолт. Если вы настроили `init.defaultBranch main` на шаге 0,
  этого блока не будет.
- `Initialized empty Git repository in .../.git/` — создана служебная папка `.git/`.
  **Весь репозиторий — это содержимое `.git/`.** Удалите её — и проект снова станет
  обычной папкой без истории.
- Обратите внимание: `git init` создал репозиторий **прямо здесь, на вашем диске** —
  это и есть тот самый **локальный** репозиторий. Никакого сервера, аккаунта или
  интернета для этого не нужно. Удалённый репозиторий на GitHub (если он вообще
  понадобится) появится позже и отдельно — в модуле 5.

Если ветка создалась как `master`, переименуем её (это нормальная, безопасная операция):

```bash
git branch -m main
```

`-m` = move/rename. Вывода нет — переименование прошло успешно.

## Шаг 2. Что внутри `.git/` (разбор служебной папки)

Прежде чем коммитить, заглянем внутрь — это и есть упражнение модуля 0.

```bash
cat .git/HEAD
```

**Вывод:**

```
ref: refs/heads/main
```

**Объяснение:** HEAD — это **указатель на текущую ветку**, а не на коммит напрямую.
Здесь написано «я сейчас на ветке `main`». Файл `refs/heads/main` пока не существует —
веток без коммитов не бывает, ветка появится вместе с первым коммитом.

```bash
ls .git
```

Ключевое внутри:

- `HEAD` — где мы сейчас (см. выше);
- `config` — конфиг этого конкретного репозитория;
- `objects/` — **база всех объектов** (blob/tree/commit), пока пустая;
- `refs/heads/` — ветки (указатели на коммиты), пока пусто.

## Шаг 3. Статус и первая постановка в индекс

```bash
git status
```

**Вывод (до `add`):**

```
On branch main

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        README.md
        index.html
        ...
nothing added to commit but untracked files present (use "git add" to track)
```

**Объяснение построчно:**

- `On branch main` — текущая ветка.
- `No commits yet` — история пуста, это будет root-commit.
- `Untracked files` — файлы, о которых Git ещё «не знает» (они есть в working tree,
  но не в index и не в истории). Это первое состояние в жизненном цикле файла:
  **untracked → staged → committed → modified**.
- В списке только исходники проекта: зависимости (`node_modules/`) мы ещё не
  устанавливали, билда (`dist/`) тоже пока нет — поэтому коммитить лишнего здесь
  нечего. К моменту, когда они появятся, мы уже будем уметь их правильно
  игнорировать (модуль 2).

Ставим всё в индекс командой `git add` — это и есть переход
**working tree → index** (первая стрелка из схемы трёх деревьев):

```bash
git add -A
git status
```

**Что значит `git add -A`:** взять **все** изменения по всему репозиторию и
синхронизировать с ними индекс — новые файлы, изменённые и удалённые. `-A` =
`--all`. Полезно знать соседние варианты:

| Команда          | Что ставит в индекс                                                                            |
| ---------------- | ---------------------------------------------------------------------------------------------- |
| `git add <файл>` | один конкретный файл                                                                           |
| `git add .`      | изменения в **текущей папке и ниже** (новые и изменённые; удаления — в новых версиях Git тоже) |
| `git add -A`     | **все** изменения во **всём** репозитории: новые, изменённые **и удалённые**                   |
| `git add -u`     | только **уже отслеживаемые** файлы (изменённые и удалённые), новые — нет                       |

Сейчас файлов мало и они все новые, поэтому `-A` нам подходит. Но в реальной работе
«добавить всё не глядя» — частый источник мусора в коммитах; в модуле 1 мы научимся
ставить изменения прицельно, по кускам (`git add -p`).

**Вывод (после `add`):**

```
On branch main

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   README.md
        new file:   index.html
        ...
```

**Объяснение:** файлы переехали из _untracked_ (working tree) в _staged_ (index) —
раздел сменился на `Changes to be committed`. Сейчас разница «не закоммичено» и
«не добавлено в индекс» наглядна: файлы добавлены в индекс, но коммита ещё нет.

> ⚠️ **Типичная ошибка:** путать «не добавлено в индекс» и «не закоммичено». Это
> два разных перехода (`add` и `commit`) через два разных дерева (index и HEAD).

## Шаг 4. Первый коммит

```bash
git commit -m "chore: initial taskflow scaffold (Vite + React + TS)"
```

**Вывод:**

```
[main (root-commit) 6061d7c] chore: initial taskflow scaffold (Vite + React + TS)
 14 files changed, 2743 insertions(+)
 create mode 100644 README.md
 create mode 100644 index.html
 ...
```

**Объяснение построчно:**

- `[main (root-commit) 6061d7c ...]` — коммит сделан в ветку `main`; `root-commit`
  значит «у него нет родителя» (самый первый коммит в истории); `6061d7c` —
  короткий SHA-1 этого коммита.
- `14 files changed, 2743 insertions(+)` — статистика: 14 файлов, 2743 добавленных строк.
- `create mode 100644 ...` — каждый файл создан с режимом `100644` (обычный
  неисполняемый файл). У исполняемых был бы `100755`.

## Шаг 5. Смотрим граф истории

```bash
git log --oneline --graph --decorate --all
```

**Вывод:**

```
* 6061d7c (HEAD -> main) chore: initial taskflow scaffold (Vite + React + TS)
```

**Объяснение флагов и вывода:**

- `--oneline` — по одному коммиту в строке (короткий хеш + заголовок).
- `--graph` — рисовать ASCII-граф (звёздочка `*` — узел-коммит; ветвления станут
  видны в модуле 3).
- `--decorate` — показывать, какие указатели смотрят на коммит.
- `--all` — показывать все ветки, а не только текущую.
- `(HEAD -> main)` — ключевой момент: **HEAD указывает на ветку `main`, а `main`
  указывает на коммит `6061d7c`**. Та самая цепочка `HEAD → ветка → commit`.

Проверим это руками:

```bash
cat .git/HEAD            # ref: refs/heads/main
cat .git/refs/heads/main # 6061d7c81e686cd29b21cb0338c866f68726c42d
```

HEAD ссылается на ветку, ветка — это просто файл с хешем коммита. Никакой магии.

## Шаг 6. Разбираем коммит руками (plumbing)

Теперь самое важное упражнение модуля — «вскрыть» объекты Git командой `cat-file`.

### 6.1. Тип и содержимое коммита

```bash
git cat-file -t HEAD     # тип объекта
git cat-file -p HEAD     # pretty-print содержимого
```

**Вывод:**

```
commit
```

```
tree 453ec3387f8046fcc6c6fb0e4b528fd3a8bed29d
author itbali <xopycaku@gmail.com> 1781976553 +0100
committer itbali <xopycaku@gmail.com> 1781976553 +0100

chore: initial taskflow scaffold (Vite + React + TS)
```

**Объяснение — из чего состоит commit-объект:**

- `tree 453ec33...` — ссылка на **корневой tree**, то есть на снимок всего проекта.
  Коммит сам по себе файлов не содержит — он указывает на дерево.
- `author` / `committer` — кто написал изменения и кто их закоммитил (обычно
  совпадают; различаются, например, после `rebase` или применения чужого патча).
  `1781976553` — Unix-время, `+0100` — часовой пояс.
- Пустая строка, затем **сообщение коммита**.
- У первого коммита **нет строки `parent`** — он корневой. У следующих появится
  `parent <hash>`, и из этих ссылок складывается граф истории.

### 6.2. Корневой tree — каталог верхнего уровня

```bash
git cat-file -p HEAD^{tree}
```

`HEAD^{tree}` — синтаксис «возьми tree, на который указывает HEAD».

**Вывод:**

```
100644 blob f9164397...  README.md
100644 blob 4fffeb29...  index.html
100644 blob 66ab9122...  package-lock.json
100644 blob 9544b62a...  package.json
040000 tree 27534dd9...  src
040000 tree d2baa171...  tests
100644 blob 18706723...  tsconfig.json
100644 blob 8b0f57b9...  vite.config.ts
```

**Объяснение — это и есть структура объектной модели:**

- Каждая строка: `режим  тип  хеш  имя`.
- Файлы (`README.md`, `package.json`, …) — это **blob**'ы.
- Папки (`src`, `tests`) — это вложенные **tree**'ы (рекурсия!).
- **Имя файла хранит tree, а не blob.** Сам blob — это только содержимое. Поэтому два
  одинаковых по содержимому файла с разными именами — это один blob и две записи в tree.

### 6.3. Спускаемся в подкаталог

```bash
git ls-tree HEAD src/
```

**Вывод:**

```
100644 blob e95b74c2...  src/App.tsx
040000 tree 2c2798db...  src/components
100644 blob b6c9e075...  src/main.tsx
040000 tree 2b635ba6...  src/state
040000 tree 753839dc...  src/styles
040000 tree 207600c6...  src/utils
```

`git ls-tree` — то же, что `cat-file -p` для tree, но удобнее для навигации.
Видно, что `src/` — это tree, внутри которого снова blob'ы и tree'ы. Так Git
рекурсивно описывает всю файловую структуру снимка.

## Шаг 7. Доказываем, что хеш — это содержимое

Главный тезис «Git адресует объекты по содержимому» легко проверить:

```bash
git rev-parse HEAD:src/utils/taskUtils.ts   # хеш blob'а файла в коммите
git hash-object README.md                   # хеш blob'а из содержимого файла на диске
```

**Вывод (пример с README):**

```
f916439718f18a311e52eae3fe38dccbe144a121
```

Сравните этот хеш со строкой `README.md` в выводе шага 6.2 — **он совпадает**
(`f9164397...`). Это и есть суть: `git hash-object` считает SHA-1 от содержимого файла
ровно тем же способом, каким Git кладёт blob в базу. Одинаковое содержимое → один и
тот же хеш → один объект в `.git/objects`. Изменили хоть один байт — хеш другой.

Заодно полезные «адресные» команды:

```bash
git rev-parse HEAD            # полный 40-символьный хеш текущего коммита
git rev-parse --short HEAD    # короткая форма (6061d7c)
```

```
6061d7c81e686cd29b21cb0338c866f68726c42d
6061d7c
```

## Шаг 8. Заглянуть в хранилище объектов

```bash
git count-objects -v
find .git/objects -type f | head
```

**Вывод:**

```
count: 22
size: 104
in-pack: 0
```

**Объяснение:**

- `count: 22` — столько «свободных» (loose) объектов сейчас в базе: blob'ы всех
  файлов + tree'ы всех каталогов + 1 commit.
- `in-pack: 0` — пока ничего не упаковано. Со временем `git gc` соберёт мелкие
  объекты в packfile для экономии места (разберём в модуле 10).
- Файлы в `.git/objects/ab/cdef...` — имя = первые 2 символа хеша (папка) + остальные
  38 (файл). Это и есть физическое хранение объектов на диске.

## Типичные ошибки модуля 0

- ❌ Думать, что Git хранит **диффы**. Git хранит **снимки**; диффы вычисляются.
- ❌ Путать «**не добавлено в индекс**» (нужен `git add`) и «**не закоммичено**»
  (нужен `git commit`). Это два перехода через два разных дерева.
- ❌ Игнорировать окончания строк на Windows → «весь файл изменился» в diff. Базовая
  защита — глобальный `core.autocrlf input` из шага 0 (на уровне репозитория закрепим
  это в модуле 2).

## Шпаргалка команд модуля 0

```bash
# настройка
git --version
git config --global user.name "..."        # автор коммитов
git config --global user.email "..."        # привязка к аккаунту
git config --global init.defaultBranch main # имя стартовой ветки

# старт репозитория
git init                      # создать .git/
git branch -m main            # переименовать ветку (при необходимости)
git status                    # состояние трёх деревьев
git add -A                    # working tree → index
git commit -m "..."           # index → HEAD (новый снимок)

# чтение истории и объектов (plumbing)
git log --oneline --graph --decorate --all
git cat-file -t <ref>         # тип объекта (commit/tree/blob)
git cat-file -p <ref>         # содержимое объекта
git cat-file -p HEAD^{tree}   # корневой tree коммита
git ls-tree HEAD <path>       # содержимое tree по пути
git rev-parse HEAD            # полный хеш
git rev-parse --short HEAD    # короткий хеш
git hash-object <file>        # посчитать хеш содержимого файла
git count-objects -v          # статистика хранилища объектов
```

Живые примеры выше прошли «насквозь» по `taskflow`: `init` → `add` → первый commit →

другом сценарии**. Чтобы не трогать первый коммит `taskflow`, эксперименты делаем в
**отдельной песочнице** — пустой папке, не связанной с курсом.

```bash
mkdir ~/git-hw0 && cd ~/git-hw0      # чистая песочница для модуля 0
git init
```

### Задание 1. Доказать «снимки, а не диффы»

Цель — своими руками увидеть, что каждый commit ссылается на **новый blob целого
файла**, а не на «разницу», и что хеш зависит только от содержимого.

**Шаги:**

1. В песочнице создайте `note.txt` со строкой `version A`, закоммитьте.
2. Поменяйте содержимое на `version B`, сделайте второй commit.
3. Сравните blob файла `note.txt` в двух коммитах:
   `git ls-tree HEAD note.txt` и `git ls-tree HEAD~1 note.txt` — хеши blob'ов **разные**.
4. Верните содержимое обратно на `version A` (не коммитя) и выполните
   `git hash-object note.txt`. Сравните с хешем blob'а из **первого** коммита.

**Критерии «сделано»:**

- В двух коммитах у `note.txt` **разные** blob-хеши (это два независимых снимка).
- `git hash-object` для возвращённого `version A` **совпадает** с blob'ом первого
  коммита — содержимое то же → хеш тот же.

**Подсказки:** `git ls-tree <commit> <file>` печатает строку вида
`100644 blob <hash>\t<file>`. Именно `<hash>` и сравнивайте.

**Самопроверка:** одинаковое содержимое → одинаковый хеш в любом репозитории мира; это и
есть «content-addressed storage».

### Задание 2. Спуститься от HEAD до конкретного blob вручную

Цель — пройти всю цепочку `HEAD → commit → tree → (под-tree) → blob`, не пользуясь
«магией», только `cat-file` и `ls-tree`.

**Шаги:**

1. В песочнице создайте вложенный файл `src/app.txt` с любым текстом, закоммитьте.
2. Посмотрите корневой tree коммита: `git cat-file -p HEAD^{tree}` — найдите строку с
   каталогом `src` (тип `tree`).
3. Провалитесь в этот под-tree по его хешу: `git cat-file -p <хеш-tree-src>` — найдите в
   нём `app.txt` (тип `blob`) и его хеш.
4. Прочитайте содержимое blob'а напрямую: `git cat-file -p <хеш-blob>` — должен
   совпасть с тем, что в файле.
5. Сверьте: `git hash-object src/app.txt` равен хешу blob'а из шага 3.

**Критерии «сделано»:**

- Дошли до хеша blob'а `app.txt`, идя только по `cat-file -p`/`ls-tree` (без `git show`).
- `git cat-file -p <blob>` печатает ровно содержимое файла.
- `git hash-object src/app.txt` совпадает с этим хешем blob'а.

**Подсказки:** `git cat-file -p HEAD^{tree}` — корневой каталог; альтернатива спуску —
`git ls-tree HEAD src/` и `git ls-tree HEAD src/app.txt`.

**Самопроверка:** вы наглядно прошли `commit → tree → tree → blob` — это и есть вся
«файловая система» Git.

### Задание 3. Наблюдать, как растёт объектная база

Цель — связать «папки и файлы» с объектами Git: каждый новый файл = blob, каждая новая
папка = tree, каждый commit = commit-объект.

**Шаги:**

1. В песочнице выполните `git count-objects -v` и запишите число `count`.
2. Добавьте **новый файл в новой папке** (например `docs/readme.md`), закоммитьте.
3. Снова выполните `git count-objects -v` и сравните `count`.
4. Объясните дельту: сколько добавилось объектов и почему (новый blob + новый tree
   для `docs/` + обновлённый корневой tree + новый commit).

**Критерии «сделано»:**

- Зафиксировали `count` до и после коммита.
- Можете назвать, какие именно объекты добавились (blob/tree/commit) и сколько.

**Подсказки:** один новый файл в новой подпапке обычно даёт **+4** объекта: blob файла,
tree подпапки, новый корневой tree и commit. Проверьте через
`git cat-file -p HEAD^{tree}`.

**Самопроверка:** «всё в Git — объект». Файлы, папки и сами коммиты — это записи в
`.git/objects`, адресуемые по хешу содержимого.
